Rappel : HyperLogLog et CompletableFuture

Cette semaine ne réexplique ni HyperLogLog (registres, zéros de tête, formule d'estimation — voir la fiche de la semaine 8) ni CompletableFuture (création, composition — voir la fiche de la semaine 9). Elle combine les deux : les traitements HyperLogLog déjà connus (ajouter, estimerCardinalite, lecture de fichiers de logs) sont simplement lancés via supplyAsync/thenApply/thenCombine au lieu d'être appelés directement.

  • HyperLogLog : structure de comptage de cardinalité approximatif — estime le nombre de valeurs distinctes vues (ex. IP uniques) avec très peu de mémoire, au prix d'une erreur d'estimation.
  • supplyAsync : démarre un calcul dans un autre thread, renvoie immédiatement un CompletableFuture<T>.
  • thenApply : enchaîne une transformation synchrone sur le résultat, sans bloquer.
  • thenCombine : fusionne les résultats de deux futures indépendantes dès que les deux sont prêtes.
Pour le détail de HyperLogLog : fiche de la semaine 8. Pour le détail de CompletableFuture (création, thenCompose, gestion d'erreurs, get/join...) : fiche de la semaine 9. Cette page ne couvre que leur combinaison.

Partie 1 - Chargement et estimation en asynchrone

Lire des fichiers est une opération d'entrée/sortie : le programme passe l'essentiel de son temps à attendre le disque. C'est le cas d'école de l'asynchrone : pendant que le chargement s'exécute dans un thread du pool, le thread principal reste libre. Le schéma est une simple chaîne :

supplyAsync(charger les accès) ──thenApply──> estimer les IP uniques ──join──> résultat

L'estimation elle-même (HyperLogLog, Hasher, AnalyseIp) ne change pas d'un iota par rapport à la semaine 8. Seule la façon de lancer le traitement change :

public CompletableFuture<List<Acces>> chargerAccesAsync(String dossier) {
    return CompletableFuture.supplyAsync(() -> {
        try {
            return analyse.chargerTousLesAcces(dossier);
        } catch (IOException e) {
            throw new RuntimeException("Erreur de lecture du dossier " + dossier, e);
        }
    });
}

public CompletableFuture<Long> estimerDepuisDossierAsync(String dossier, HyperLogLog hyperLogLog) {
    return chargerAccesAsync(dossier)
        .thenApply(acces -> AnalyseIp.estimerIpUniques(acces, hyperLogLog));
}

Utilisation :

Long estimation = analyseAsync
    .estimerDepuisDossierAsync("logs", new HyperLogLog(hasher, 4))
    .join();   // on ne bloque qu'au moment où on a vraiment besoin du résultat

Le piège des exceptions checked

Le Supplier<T> attendu par supplyAsync ne déclare aucun throws. Une lambda qui appelle une méthode levant une exception checked (comme IOException pour la lecture de fichier) ne compile donc pas telle quelle.

Impossible de faire : 
// NE COMPILE PAS : IOException est checked, Supplier ne la déclare pas
CompletableFuture.supplyAsync(() -> analyse.chargerTousLesAcces(dossier));

La solution : attraper l'exception checked à l'intérieur de la lambda et la relancer sous forme d'exception unchecked (RuntimeException), qui n'a pas besoin d'être déclarée :

CompletableFuture.supplyAsync(() -> {
    try {
        return analyse.chargerTousLesAcces(dossier);
    } catch (IOException e) {
        throw new RuntimeException("Erreur de lecture du dossier " + dossier, e);
    }
});
Toujours passer la cause (new RuntimeException(message, e)) plutôt que d'afficher la trace et de continuer : sinon l'appelant de join() perd complètement l'information de ce qui a échoué.

Mesurer séquentiel vs asynchrone

Pour objectiver le gain, on chronomètre les deux approches sur le même jeu de dossiers : la version séquentielle traite un dossier après l'autre, la version asynchrone lance tous les chargements en parallèle avant de bloquer sur le résultat.

long debutSeq = System.currentTimeMillis();
List<Long> resultatsSeq = analyseAsync.estimerPlusieursDossiersSequentiellement(dossiers, hasher);
long dureeSeq = System.currentTimeMillis() - debutSeq;

long debutAsync = System.currentTimeMillis();
List<Long> resultatsAsync = analyseAsync.estimerPlusieursDossiersAsync(dossiers, hasher).join();
long dureeAsync = System.currentTimeMillis() - debutAsync;

System.out.println("Séquentiel : " + dureeSeq + " ms");
System.out.println("Asynchrone : " + dureeAsync + " ms");

Pour traiter plusieurs dossiers en parallèle puis attendre tous les résultats d'un coup, on combine allOf (qui attend que toutes les futures soient terminées) et une relecture des résultats déjà disponibles via join() (qui ne bloque alors plus, puisque tout est déjà fini) :

public CompletableFuture<List<Long>> estimerPlusieursDossiersAsync(List<String> dossiers, Hasher hasher) {
    List<CompletableFuture<Long>> futures = dossiers.stream()
        .map(dossier -> estimerDepuisDossierAsync(dossier, new HyperLogLog(hasher, 4)))
        .toList();

    return CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]))
        .thenApply(v -> futures.stream().map(CompletableFuture::join).toList());
}
Le gain n'apparaît que si le travail est réellement parallélisable (plusieurs fichiers/dossiers indépendants) et dominé par de l'attente d'entrée/sortie. Sur un seul petit fichier, l'overhead de création des threads peut rendre l'asynchrone plus lent que le séquentiel.

Partie 2 - Fusionner deux HyperLogLog

Un HyperLogLog n'est qu'un tableau de registres, chacun retenant le maximum de zéros de tête observé pour les valeurs qui lui sont tombées dessus. Conséquence directe : pour fusionner deux estimateurs de même taille ayant vu des ensembles de valeurs différents, il suffit de prendre, registre par registre, le maximum des deux :

fusion.registres[i] = max(premier.registres[i], second.registres[i])

HyperLogLog.fusionner

public void fusionner(HyperLogLog autre) 
public void fusionner(HyperLogLog autre) {
    Util.checkObject(autre);
    if (autre.registres.length != registres.length)
        throw new IllegalArgumentException(
            "Les deux HyperLogLog doivent avoir le même nombre de registres");
    for (int i = 0; i < registres.length; i++) {
        registres[i] = Math.max(registres[i], autre.registres[i]);
    }
}

Le tableau obtenu est exactement celui qu'on aurait eu en ajoutant toutes les valeurs des deux ensembles dans un seul estimateur — peu importe quel estimateur a vu quelle valeur, ni combien de fois. Deux propriétés en découlent :

  • Commutativité : fusionner(a, b) et fusionner(b, a) donnent le même tableau.
  • Associativité : fusionner dans n'importe quel ordre de regroupement donne le même tableau.

Un CompletableFuture par routeur (scatter/gather)

Chaque source de données (ici, un fichier de logs par routeur réseau) construit son propre petit estimateur dans son coin — donc dans son propre thread — et les résultats se combinent à la fin, sans perte de précision. C'est exactement ainsi que les bases de données distribuées (Redis PFMERGE/PFCOUNT, BigQuery, Redshift...) comptent des cardinalités réparties sur des dizaines de machines.

Le schéma est un éventail (scatter/gather) plutôt qu'une chaîne :

                 ┌─ supplyAsync(HLL du routeur 0) ─┐
index des        ├─ supplyAsync(HLL du routeur 1) ─┤   fusion des résultats
fichiers ──────> ├─ supplyAsync(HLL du routeur 2) ─┼─> (thenCombine en cascade
                 ├─ ...                            │    ou allOf puis boucle)
                 └─ supplyAsync(HLL du routeur N) ─┘

Un CompletableFuture<HyperLogLog> par fichier routeur (lecture + construction d'un estimateur local) :

// un HyperLogLog local pour un fichier routeur
public HyperLogLog hllPourFichier(String dossier, String nomFichier) throws IOException {
    HyperLogLog hyperLogLog = new HyperLogLog(hasher, NB_BITS_INDEX);
    analyse.chargerAcces(dossier + "/" + nomFichier).stream()
        .map(Acces::getIp)
        .forEach(hyperLogLog::ajouter);
    return hyperLogLog;
}

// la version asynchrone : mêmes pièges checked que plus haut
public CompletableFuture<HyperLogLog> hllPourFichierAsync(String dossier, String nomFichier) {
    return CompletableFuture.supplyAsync(() -> {
        try {
            return hllPourFichier(dossier, nomFichier);
        } catch (IOException e) {
            throw new RuntimeException("Erreur de lecture du fichier " + nomFichier, e);
        }
    });
}

Pour rassembler, thenCombine combine deux futures dès qu'elles sont toutes les deux complétées. On chaîne un thenCombine par fichier, en partant d'une future déjà complétée comme point de départ neutre :

public CompletableFuture<HyperLogLog> fusionnerTousLesRouteursAsync(String dossier) throws IOException {
    CompletableFuture<HyperLogLog> resultat =
        CompletableFuture.completedFuture(new HyperLogLog(hasher, NB_BITS_INDEX));
    for (String nomFichier : analyse.listerFichiers(dossier)) {
        resultat = resultat.thenCombine(hllPourFichierAsync(dossier, nomFichier),
            (fusion, hllRouteur) -> {
                fusion.fusionner(hllRouteur);
                return fusion;
            });
    }
    return resultat;
}
CompletableFuture.completedFuture(valeur) crée une future déjà complétée — utile comme accumulateur initial d'une cascade de thenCombine, ici un estimateur vide.

Pourquoi le résultat est déterministe

Avec des threads, l'ordre d'exécution est imprévisible : le routeur 7 peut finir avant le routeur 2 aujourd'hui, et après lui demain. En général, c'est la porte ouverte aux résultats non reproductibles (les race conditions de la semaine 9).

Ici, non : la fusion étant commutative et associative, l'ordre et le regroupement des fusions n'ont aucune influence sur le tableau final. Le résultat asynchrone est identique au résultat séquentiel, à chaque exécution.

Critère de conception à retenir : les traitements parallèles les plus sûrs sont ceux dont l'étape de combinaison est insensible à l'ordre (commutative et associative) — comme un maximum, une somme, une union d'ensembles. C'est ce qui rend ce traitement parfaitement testable en JUnit, sans aléa, et réutilisable pour toute agrégation parallèle.

Pièges fréquents

Problème Cause probable Correction
Rien ne s'affiche, ou NullPointerException sur le résultat Le résultat de la chaîne asynchrone est lu avant que join()/get() n'ait été appelé. Appeler join() (ou get()) en bout de chaîne, uniquement quand le résultat est vraiment nécessaire.
Erreur de compilation dans la lambda de supplyAsync Une exception checked (IOException) est levée dans une lambda alors que Supplier ne déclare aucun throws. Attraper l'exception dans la lambda et la relancer en RuntimeException(message, e).
On pense qu'il faut fusionner « dans le bon ordre » Confusion avec les race conditions de la semaine 9 : ici la fusion est une opération de max, commutative et associative. Aucun ordre particulier requis — le résultat est identique quel que soit l'ordre d'arrivée des thenCombine.
Estimation faussée, valeurs perdues sous charge Un seul HyperLogLog partagé entre plusieurs threads, modifié en parallèle par ajouter sans synchronisation. Un HyperLogLog local par tâche asynchrone, puis fusionner les résultats une fois chaque tâche terminée.
Le programme se termine avant d'avoir tout traité Le main ne bloque jamais sur la future finale (threads du pool commun daemon). join() sur la future racine avant de quitter le main.

Référence rapide

// Chaîne asynchrone : charger puis estimer (partie 1)
CompletableFuture<List<Acces>> futureAcces = CompletableFuture.supplyAsync(() -> {
    try {
        return analyse.chargerTousLesAcces(dossier);
    } catch (IOException e) {
        throw new RuntimeException("Erreur de lecture", e);
    }
});
CompletableFuture<Long> futureEstimation =
    futureAcces.thenApply(acces -> AnalyseIp.estimerIpUniques(acces, hyperLogLog));
long estimation = futureEstimation.join();   // bloque en tout dernier

// Fusion de deux HyperLogLog (partie 2)
hyperLogLogA.fusionner(hyperLogLogB);   // max registre par registre, commutatif et associatif

// Scatter/gather : un CompletableFuture par fichier, fusionnés en cascade
CompletableFuture<HyperLogLog> resultat =
    CompletableFuture.completedFuture(new HyperLogLog(hasher, 4));
for (String nomFichier : fichiers) {
    resultat = resultat.thenCombine(hllPourFichierAsync(dossier, nomFichier),
        (fusion, hllRouteur) -> { fusion.fusionner(hllRouteur); return fusion; });
}
HyperLogLog hllGlobal = resultat.join();
long cardinaliteTotale = hllGlobal.estimerCardinalite();
