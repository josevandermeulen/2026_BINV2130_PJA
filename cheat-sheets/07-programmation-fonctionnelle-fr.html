Interfaces fonctionnelles

Une interface fonctionnelle est une interface qui ne déclare qu'une seule méthode abstraite. C'est ce qui permet à une expression lambda de représenter une implémentation de cette interface : la lambda fournit le corps de l'unique méthode abstraite.

@FunctionalInterface

@FunctionalInterface
public interface Predicate<T> {
    boolean test(T t);
}

Cette annotation est facultative mais recommandée : elle demande au compilateur de vérifier que l'interface possède bien une seule méthode abstraite (elle peut en revanche avoir des méthodes default ou static, voir plus bas).

Java 8 fournit dans le package java.util.function un ensemble d'interfaces fonctionnelles prêtes à l'emploi, couvrant la plupart des cas courants :

Interface Méthode abstraite Rôle Utilisée par
Predicate<T> boolean test(T t) Teste une condition sur une valeur. filter
Function<T, R> R apply(T t) Transforme une valeur de type T en une valeur de type R. map
BiFunction<T, U, R> R apply(T t, U u) Combine deux valeurs en une seule. Map.merge, reduce
Consumer<T> void accept(T t) Consomme une valeur sans rien renvoyer (effet de bord). forEach
BiConsumer<T, U> void accept(T t, U u) Consomme deux valeurs sans rien renvoyer. Map.forEach
Supplier<T> T get() Fournit une valeur, sans paramètre. orElseGet
UnaryOperator<T> T apply(T t) Function<T, T> spécialisée : entrée et sortie de même type. List.replaceAll
BinaryOperator<T> T apply(T t1, T t2) Combine deux valeurs du même type en une seule du même type. reduce
Comparator<T> int compare(T o1, T o2) Compare deux valeurs pour établir un ordre. sorted
Runnable void run() Exécute une action, sans paramètre ni retour. Threads, tâches planifiées
N'importe quelle interface à une seule méthode abstraite peut servir d'interface fonctionnelle, y compris une interface qu'on écrit soi-même (voir l'exemple EmployeNomFunction ci-dessous).

Expressions lambda

Une expression lambda est une fonction anonyme : elle représente l'implémentation de l'unique méthode abstraite d'une interface fonctionnelle. C'est un raccourci syntaxique pour créer et instancier une classe qui implémente cette interface.

Syntaxe

// Sans paramètre
() -> 42

// Un seul paramètre (les parenthèses sont optionnelles)
e -> e.getNom()
(e) -> e.getNom()

// Plusieurs paramètres
(e1, e2) -> e1.getTaille() - e2.getTaille()

// Type explicite des paramètres
(Employe e1, Employe e2) -> e1.getTaille() - e2.getTaille()

// Corps avec plusieurs instructions : accolades + return explicite
(e1, e2) -> {
    if (e1.getTaille() == e2.getTaille()) {
        return e1.getNom().compareTo(e2.getNom());
    }
    return e2.getTaille() - e1.getTaille();
}

Une lambda est une implémentation implicite

Passer une lambda là où une interface fonctionnelle est attendue équivaut à écrire une classe qui l'implémente et à l'instancier.

// Avec une lambda
List<Employe> hommes = employes.stream()
    .filter(e -> e.getGenre() == Genre.HOMME)
    .toList();

// Équivalent explicite, sans lambda
public class PredicatGenreHomme implements Predicate<Employe> {
    @Override
    public boolean test(Employe e) {
        return e.getGenre() == Genre.HOMME;
    }
}

List<Employe> hommesBis = employes.stream()
    .filter(new PredicatGenreHomme())
    .toList();

La méthode filter appelle en réalité test() sur chaque élément du stream ; la lambda n'est qu'un sucre syntaxique pour cette implémentation.

Les lambdas hors des Streams

Une lambda peut être utilisée partout où une interface fonctionnelle est attendue, pas seulement dans un pipeline Stream : un Comparator passé à List.sort, un Runnable passé à un Thread, etc.

List<Employe> tries = new ArrayList<>(employes);
tries.sort((e1, e2) -> e1.getNom().compareTo(e2.getNom()));

Références de méthode

Quand le corps d'une lambda se contente d'appeler une méthode existante, on peut la remplacer par une référence de méthode, plus concise. Il en existe quatre formes.

Forme Syntaxe Exemple Équivalent lambda
Méthode statique Classe::methodeStatique Integer::parseInt s -> Integer.parseInt(s)
Méthode d'instance d'un objet particulier instance::methode System.out::println x -> System.out.println(x)
Méthode d'instance d'un objet quelconque d'un type donné Classe::methode Employe::getNom e -> e.getNom()
Référence de constructeur Classe::new ArrayList::new () -> new ArrayList<>() 
// Méthode d'instance d'un objet quelconque du type Employe
List<String> noms = employes.stream()
    .map(Employe::getNom)
    .toList();

// Méthode statique
List<Integer> entiers = chaines.stream()
    .map(Integer::parseInt)
    .toList();

// Référence de constructeur : produit un nouveau conteneur
List<Employe> copie = employes.stream()
    .collect(Collectors.toCollection(ArrayList::new));
Une référence de méthode n'est pas un appel de méthode : Employe::getNom désigne la méthode elle-même, à appliquer plus tard sur un employé donné par le code qui la reçoit (ici, map).

Predicate

Classe : java.util.function.Predicate<T>. Représente un test booléen sur une valeur de type T.

Méthode abstraite

boolean test(T t) 
Predicate<Integer> estPositif = n -> n > 0;
boolean resultat = estPositif.test(5); // true

Combiner des Predicate

Predicate<T> and(Predicate<? super T> other) Predicate<T> or(Predicate<? super T> other) Predicate<T> negate() 
Predicate<Integer> estPositif = n -> n > 0;
Predicate<Integer> estPair = n -> n % 2 == 0;

Predicate<Integer> positifEtPair = estPositif.and(estPair);
Predicate<Integer> positifOuPair = estPositif.or(estPair);
Predicate<Integer> estNegatif = estPositif.negate();

Ces méthodes default permettent de composer des prédicats sans écrire de nouvelle lambda pour chaque combinaison.

Sans Stream : utiliser test directement

public static <T> List<T> allMatches(List<T> list, Predicate<T> match) {
    List<T> result = new ArrayList<>();
    for (T element : list) {
        if (match.test(element)) {
            result.add(element);
        }
    }
    return result;
}

// Utilisation
List<Integer> pairs = allMatches(nombres, n -> n % 2 == 0);

Un Predicate n'a d'intérêt que dans un contexte qui l'appelle via test() ; en dehors d'un Stream, on écrit soi-même la boucle qui l'utilise.

Function et BiFunction

Classe : java.util.function.Function<T, R>. Transforme une valeur de type T en une valeur de type R.

Méthode abstraite

R apply(T t) 
Function<Employe, String> nomEmploye = e -> e.getNom();
String nom = nomEmploye.apply(employe);

// Sous forme de classe explicite
public class EmployeNomFunction implements Function<Employe, String> {
    @Override
    public String apply(Employe employe) {
        return employe.getNom();
    }
}

Chaîner des Function

<V> Function<T, V> andThen(Function<? super R, ? extends V> after) <V> Function<V, R> compose(Function<? super V, ? extends T> before) 
Function<String, Integer> longueur = String::length;
Function<Integer, String> versChaine = n -> "Longueur : " + n;

// andThen : applique this, puis after
Function<String, String> combinee = longueur.andThen(versChaine);
combinee.apply("Java"); // "Longueur : 4"

BiFunction

R apply(T t, U u) 
BiFunction<Integer, Integer, Integer> addition = (a, b) -> a + b;
int somme = addition.apply(3, 4); // 7

Utile pour combiner deux valeurs, par exemple dans Map.merge(clé, valeur, BiFunction).

Sans Stream : transformAll

public static <P, R> List<R> transformAll(List<P> list, Function<P, R> transform) {
    List<R> result = new ArrayList<>();
    for (P element : list) {
        result.add(transform.apply(element));
    }
    return result;
}

Consumer et Supplier

Consumer<T>

void accept(T t) Consumer<T> andThen(Consumer<? super T> after)

Consomme une valeur sans rien renvoyer : sert typiquement à produire un effet de bord (affichage, ajout dans une collection).

Consumer<Employe> afficher = e -> System.out.println(e.getNom());
employes.forEach(afficher);

// Sous forme de classe explicite, avec état
public class EmployeeConsumer implements Consumer<Employe> {
    private final List<String> noms;

    public EmployeeConsumer(List<String> noms) {
        this.noms = noms;
    }

    @Override
    public void accept(Employe employe) {
        System.out.println(employe);
        noms.add(employe.getNom());
    }
}

Supplier<T>

T get()

Fournit une valeur, sans paramètre : sert à différer un calcul ou une lecture (fichier, base de données) jusqu'au moment où la valeur est réellement nécessaire.

Supplier<String> valeurParDefaut = () -> "Inconnu";
String nom = optNom.orElseGet(valeurParDefaut);

Supplier<List<String>> nouvelleListe = ArrayList::new;
List<String> liste = nouvelleListe.get();
orElseGet(Supplier) n'exécute le calcul que si l'Optional est vide, contrairement à orElse(valeur) qui évalue toujours son argument. Préférer orElseGet quand la valeur par défaut est coûteuse à construire (lecture d'un fichier, par exemple).

Comparateurs fonctionnels

Classe : java.util.Comparator<T>. Interface fonctionnelle à une méthode abstraite compare, enrichie de nombreuses méthodes default/static pour construire des comparateurs par composition plutôt qu'en écrivant une classe.

Construire un comparateur à partir d'une clé

static <T, U extends Comparable<? super U>> Comparator<T> comparing(Function<? super T, ? extends U> keyExtractor) static <T> Comparator<T> comparingInt(ToIntFunction<? super T> keyExtractor) static <T> Comparator<T> comparingDouble(ToDoubleFunction<? super T> keyExtractor) 
// Trier par nom (ordre naturel de String)
List<Employe> parNom = employes.stream()
    .sorted(Comparator.comparing(Employe::getNom))
    .toList();

// Trier par taille (int) : évite le boxing
List<Employe> parTaille = employes.stream()
    .sorted(Comparator.comparingInt(Employe::getTaille))
    .toList();

Inverser l'ordre

Comparator<T> reversed() 
List<Employe> parTailleDecroissante = employes.stream()
    .sorted(Comparator.comparingInt(Employe::getTaille).reversed())
    .toList();

Critère de départage : thenComparing

Comparator<T> thenComparing(Comparator<? super T> other) Comparator<T> thenComparing(Function<? super T, ? extends U> keyExtractor) 
// Par taille décroissante, puis par nom en cas d'égalité
Comparator<Employe> comparateur = Comparator
    .comparingInt(Employe::getTaille).reversed()
    .thenComparing(Employe::getNom);

List<Employe> tries = new ArrayList<>(employes);
tries.sort(comparateur);

// Equivalent écrit "à la main" avec une lambda
tries.sort((e1, e2) -> {
    if (e1.getTaille() == e2.getTaille()) {
        return e1.getNom().compareTo(e2.getNom());
    }
    return e2.getTaille() - e1.getTaille();
});

thenComparing est le pendant fonctionnel du if (égalité) { ... } qu'on écrirait dans un Comparator manuel : il s'applique seulement quand le comparateur précédent renvoie 0.

Ordres prédéfinis

static <T extends Comparable<? super T>> Comparator<T> naturalOrder() static <T extends Comparable<? super T>> Comparator<T> reverseOrder() 
List<Integer> tries = nombres.stream()
    .sorted(Comparator.naturalOrder())
    .toList();

flatMap

La méthode flatMap aplatit une structure de données hiérarchique : là où map produirait un stream d'éléments qui sont eux-mêmes des collections (Stream<List<T>>), flatMap fusionne le stream produit pour chaque élément en un seul stream plat (Stream<T>).

Signature

Stream<R> flatMap(Function<? super T, ? extends Stream<? extends R>> mapper) 
// Un Portfolio contient un Trader et une liste d'actions détenues
class Portfolio {
    private final Trader trader;
    private final List<String> actions;
    // getters ...
}

List<Portfolio> portfolios = ...;

// Toutes les actions de tous les portefeuilles (avec doublons)
List<String> actions = portfolios.stream()
    .flatMap(portfolio -> portfolio.getActions().stream())
    .toList();
Le réflexe : dès que le modèle de données est hiérarchique (une liste d'objets contenant chacun une liste), map seul renvoie un Stream de listes ; flatMap avec .stream() sur le champ imbriqué aplatit le tout en un seul niveau.

flatMap combiné à map, distinct, sorted

// Actions uniques avec leur prix, triées par prix croissant (puis par symbole)
List<Map.Entry<String, Double>> resultat = portfolios.stream()
    .flatMap(portfolio -> portfolio.getActions().stream())
    .distinct()
    .filter(prices::containsKey)
    .map(symbole -> Map.entry(symbole, prices.get(symbole)))
    .sorted(Comparator.comparingDouble((Map.Entry<String, Double> e) -> e.getValue())
            .thenComparing(Map.Entry::getKey))
    .toList();

Map.entry(clé, valeur) construit un couple clé/valeur immuable, pratique pour manipuler des paires sans créer une classe dédiée.

flatMap produisant des paires

// Un couple (action, trader) par action détenue
List<Map.Entry<String, Trader>> couples = portfolios.stream()
    .flatMap(portfolio -> portfolio.getActions().stream()
            .map(action -> Map.entry(action, portfolio.getTrader())))
    .toList();

// Regrouper : pour chaque action, la liste des traders qui la possèdent
Map<String, List<Trader>> parAction = couples.stream()
    .collect(Collectors.groupingBy(
        Map.Entry::getKey,
        Collectors.mapping(Map.Entry::getValue, Collectors.toList())
    ));

Ici, le flatMap lui-même contient un map imbriqué : pour chaque portefeuille, on transforme chaque action détenue en un couple (action, trader), puis on aplatit tous ces couples en un seul stream.

Méthodes default et static

Avant Java 8, une interface ne pouvait déclarer que des méthodes abstraites. Ajouter une nouvelle méthode à une interface obligeait donc à modifier toutes ses implémentations existantes — un problème majeur pour faire évoluer des interfaces comme Collection sans casser le code existant.

Méthodes default

Une méthode default fournit une implémentation directement dans l'interface. Les classes qui implémentent l'interface en héritent automatiquement, sans devoir l'écrire elles-mêmes (mais peuvent la redéfinir si besoin).

public interface List<E> extends Collection<E> {
    default void sort(Comparator<? super E> c) {
        // implémentation fournie par l'interface elle-même
    }
}

C'est ainsi que stream() a pu être ajoutée à Collection en Java 8 sans casser aucune classe existante qui l'implémentait déjà : toutes héritent de l'implémentation par défaut.

Méthodes static

Une interface peut aussi déclarer des méthodes statiques, appelées sur l'interface elle-même plutôt que sur une instance — comme des méthodes utilitaires liées au type.

public interface Comparator<T> {
    static <T extends Comparable<? super T>> Comparator<T> naturalOrder() {
        // ...
    }
}

Comparator<String> c = Comparator.naturalOrder();

Rappel : les generics dans une signature d'interface

La signature default void sort(Comparator<? super E> c) se lit ainsi : c est un Comparator capable de comparer des objets de type E (le type générique de la liste) ou de l'une de ses super-classes. Cette flexibilité (? super E) permet, par exemple, de trier une List<Dog> avec un Comparator<Animal>.

Immuabilité

Un objet immuable ne peut pas changer d'état après sa construction. C'est un pilier du design fonctionnel : sans mutation partagée, un objet peut être passé librement entre lambdas, threads ou méthodes sans risque d'effet de bord inattendu.

Règles pour une classe immuable

  • Tous les champs sont private final.
  • Aucun setter : les valeurs sont fixées uniquement dans le constructeur.
  • La classe n'est pas extendsible de façon incontrôlée (idéalement final), pour éviter qu'une sous-classe n'introduise de la mutabilité.
  • Si un champ est une référence vers un objet mutable (une List, un Date, un tableau), il faut en faire une copie défensive — à la construction et/ou dans le getter — pour empêcher l'appelant de modifier l'état interne via la référence obtenue.

Exemple : une classe pas encore immuable

public class Portfolio {
    private final Trader trader;
    private final List<String> actions;

    public Portfolio(Trader trader, List<String> actions) {
        this.trader = trader;
        this.actions = actions; // référence partagée avec l'appelant !
    }

    public List<String> getActions() {
        return actions; // expose directement la liste interne
    }
}

// Le problème :
List<String> mesActions = new ArrayList<>(List.of("AAPL", "TSLA"));
Portfolio p = new Portfolio(trader, mesActions);
mesActions.add("GOOGL");     // modifie discrètement le Portfolio !
p.getActions().add("MSFT"); // modifie aussi le Portfolio, de l'extérieur

Version immuable, avec copies défensives

public final class Portfolio {
    private final Trader trader;
    private final List<String> actions;

    public Portfolio(Trader trader, List<String> actions) {
        this.trader = trader;
        this.actions = List.copyOf(actions); // copie non modifiable
    }

    public List<String> getActions() {
        return actions; // déjà non modifiable, sûr à exposer tel quel
    }
}

List.copyOf(...) (comme List.of(...)) renvoie une liste immuable : toute tentative de modification lève une UnsupportedOperationException. Alternative équivalente : Collections.unmodifiableList(new ArrayList<>(actions)).

Modifier un objet immuable ne se fait jamais « sur place » : on construit un nouvel objet avec l'état voulu. C'est exactement le principe des Streams : chaque opération intermédiaire renvoie un nouveau Stream sans modifier la source.

Exceptions en style fonctionnel

Les méthodes abstraites des interfaces fonctionnelles standard (Function.apply, Predicate.test, Consumer.accept...) ne déclarent aucune exception vérifiée (checked exception). Il est donc impossible d'appeler directement, dans le corps d'une lambda, une méthode qui déclare throws IOException ou une autre exception vérifiée.

Le problème

// Ne compile pas : IOException est vérifiée, Function.apply() ne la déclare pas
Function<String, List<String>> lireFichier = chemin -> Files.readAllLines(Path.of(chemin));

Solution 1 : attraper l'exception et la relancer en non vérifiée

Function<String, List<String>> lireFichier = chemin -> {
    try {
        return Files.readAllLines(Path.of(chemin));
    } catch (IOException e) {
        throw new RuntimeException("Lecture impossible : " + chemin, e);
    }
};

C'est la solution la plus courante : on convertit l'exception vérifiée en exception non vérifiée (RuntimeException), qui n'a pas besoin d'être déclarée dans la signature de la méthode fonctionnelle. La cause originale (e) est conservée en second argument, pour ne pas perdre l'information de débogage.

Solution 2 : gérer l'erreur localement, sans propager

// Ignorer une ligne invalide plutôt que d'interrompre tout le traitement
List<Acces> acces = new ArrayList<>();
for (String ligne : lignes) {
    try {
        acces.add(parserLigne(ligne));
    } catch (IllegalArgumentException e) {
        System.out.println("Ligne ignorée : " + ligne);
    }
}

Quand l'erreur ne remet pas en cause tout le pipeline (une seule ligne invalide parmi beaucoup d'autres valides), on l'attrape au plus près de sa source et on continue, plutôt que de la propager jusqu'à l'appelant.

Solution 3 : une interface fonctionnelle personnalisée

@FunctionalInterface
public interface FonctionAvecException<T, R> {
    R apply(T t) throws IOException;
}

Cette solution évite d'avaler l'exception dans la lambda, mais elle exige d'écrire sa propre interface fonctionnelle (les interfaces standard de java.util.function ne peuvent pas être modifiées) et complique l'utilisation directe avec l'API Stream, dont les méthodes attendent les interfaces standard.

Une exception vérifiée avalée silencieusement (un catch vide) masque un vrai problème. Toujours au minimum journaliser l'erreur, ou la relancer sous une forme non vérifiée avec la cause d'origine conservée.

Streams parallèles

Les opérations d'un stream ne modifient jamais le stream d'origine : traiter les éléments dans un ordre différent, voire simultanément sur plusieurs cœurs, ne change donc pas le résultat. L'API Stream fournit une version parallèle prête à l'emploi.

liste.parallelStream()
     .filter(x -> x % 2 == 0)
     .map(x -> x * x)
     .toList();

// ou, à partir d'un stream séquentiel déjà obtenu :
liste.stream().parallel()...

Un stream parallèle répartit les éléments de la collection entre plusieurs threads (via le ForkJoinPool commun) puis combine les résultats partiels.

La parallélisation n'est rentable que sur de gros volumes de données et des opérations coûteuses par élément — sur une petite collection, le surcoût de répartition/combinaison dépasse le gain. Elle exige aussi que les opérations soient sans effet de bord (pas de modification d'un état partagé dans un forEach/map), sous peine de race condition.

Pièges fréquents

Problème Cause probable Correction
Erreur de compilation : une lambda ne compile pas avec une méthode qui lève une exception vérifiée. La méthode abstraite de l'interface fonctionnelle standard utilisée (Function, Predicate...) ne déclare pas cette exception. Attraper l'exception dans le corps de la lambda et la relancer en RuntimeException, ou écrire une interface fonctionnelle personnalisée.
Utiliser map à la place de flatMap sur une structure hiérarchique. map(p -> p.getActions()) renvoie un Stream<List<String>>, pas un Stream<String>. Utiliser flatMap(p -> p.getActions().stream()) pour aplatir.
NullPointerException lors de l'appel d'une méthode sur un champ censé être immuable. Le constructeur a stocké la référence reçue sans copie défensive ; l'appelant l'a modifiée (voire mise à null) après coup. Copier la structure dans le constructeur (List.copyOf(...)) plutôt que d'en stocker la référence brute.
Le tri avec thenComparing ne départage pas comme attendu. Le premier Comparator ne renvoie jamais 0 pour les éléments concernés (par exemple, comparaison sur une clé toujours différente). Vérifier que le critère principal peut effectivement produire des égalités ; sinon thenComparing n'est jamais atteint.
Confondre Function<T, R> et Consumer<T>. Function renvoie une valeur (apply), Consumer ne renvoie rien (accept) — souvent utilisé par erreur pour transformer une valeur. Utiliser Function avec map pour transformer, Consumer avec forEach pour un effet de bord.
Une classe censée être immuable est en réalité modifiable de l'extérieur. Le getter renvoie directement la référence interne d'une collection mutable. Renvoyer une collection non modifiable (List.copyOf(...) à la construction suffit si le getter renvoie ensuite cette même référence).
orElse(calculCouteux()) ralentit le programme même quand l'Optional est présent. orElse évalue toujours son argument, contrairement à orElseGet. Utiliser orElseGet(() -> calculCouteux()), qui n'exécute le Supplier que si nécessaire.
