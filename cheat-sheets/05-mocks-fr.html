Contexte d'utilisation

On parle de vrai test unitaire quand on teste une seule classe sans tester ses dépendances : on isole la classe testée en remplaçant ses collaborateurs par des doublures de test. Si l'on teste le fonctionnement de plusieurs classes qui communiquent réellement entre elles, on parle de test d'intégration.

Quand la classe testée dépend d'autres objets, un test unitaire doit pouvoir l'isoler : on remplace ses dépendances par des doublures de test dont on contrôle entièrement le comportement. Deux approches :

  • Le stub : une petite classe écrite à la main qui implémente l'interface de la dépendance et retourne des valeurs fixes.
  • Le mock : un objet simulé généré par une bibliothèque comme Mockito, configurable appel par appel, et dont on peut vérifier les interactions.

Le critère qui distingue vraiment les deux n'est pas la technique utilisée pour les créer (main vs bibliothèque), mais ce que le test vérifie :

  • Stub → teste l'état. Le test appelle la méthode de l'objet testé, puis vérifie par un simple assert le résultat produit (valeur de retour, état de l'objet) — peu importe comment l'objet testé a interagi avec le stub. Exemple : Calculator.addToX(3, 4) avec un DataServiceStub qui renvoie toujours 5 ; le test vérifie seulement assertEquals(12, result).
  • Mock → teste le comportement. Le test vérifie que l'objet testé a effectué les bons appels sur ses dépendances (bonne méthode, bons arguments, bon nombre de fois) ; c'est le mock lui-même qui échoue le test si l'appel attendu n'a pas eu lieu. Exemple : vérifier qu'un service de facturation a bien appelé dao.save(facture), via Mockito.verify(dao).save(facture).

Dépendance Maven : org.mockito:mockito-junit-jupiter (elle amène Mockito Core).

Un mock ne s'utilise que pour les dépendances de la classe testée — jamais pour la classe testée elle-même. Si l'objet est trivial à construire (un simple objet de données), il est souvent plus simple de l'instancier directement.

Taxonomie des doublures de test

Le terme générique pour tout objet qui remplace une vraie dépendance dans un test est doublure de test (« test double »). Ce cours en utilise principalement deux, mais la famille complète compte quatre grandes catégories :

Type Définition conceptuelle
Fake object Implémentation fonctionnelle simplifiée d'une dépendance, qui prend un raccourci la rendant inadaptée à la production (ex. une base de données en mémoire à la place d'une vraie base de données).
Stub Faux objet « sans intelligence » : aucune logique, aucun calcul à partir des arguments reçus. Configuré à l'avance pour renvoyer une réponse fixe. Sert à tester l'état.
Mock Objet configuré avec des attentes précises sur les appels qu'il doit recevoir (méthode, arguments, ordre, nombre) ; c'est lui qui fait échouer le test si un appel attendu n'a pas eu lieu. Sert à tester le comportement.
Spy Un stub qui enregistre en plus les informations sur la façon dont il a été appelé (nombre d'appels, paramètres reçus, …), afin de les vérifier après coup dans le test.
Le Spy de Mockito (voir plus bas, section « Mockito - bases ») correspond à cette définition : il encapsule un vrai objet, en appelle réellement les méthodes par défaut, et permet de vérifier après coup les interactions — comme un stub augmenté d'un enregistrement des appels.

(Référence : xUnit Test Patterns, Gerard Meszaros, http://xunitpatterns.com/Test%20Double.html)

Cycles de vie d'un test (modèle AAA)

Le modèle AAA (Arrange / Act / Assert) se décline différemment selon qu'on utilise un stub ou un mock, car la vérification ne se fait pas au même endroit.

Avec un stub

  • Arrange – Set context : préparez l'objet sous test, ses collaborateurs « stubs » et connectez-les.
  • Act : appelez la méthode à tester.
  • Interact : l'objet sous test fait son travail en collaborant avec les stubs.
  • Assert : vérifiez l'état de l'objet sous test (valeur de retour, état après l'appel) — l'assertion vit dans le code du test.

Avec un mock

  • Arrange – Set context : préparez l'objet sous test.
  • Arrange – Configure the mock : indiquez au mock quoi attendre (méthodes, arguments).
  • Act : appelez la méthode à tester.
  • Interact : l'objet sous test collabore avec le mock.
  • Mock remembers its interactions : le mock retient automatiquement les appels reçus.
  • Assert expectations : vérifiez, via Mockito.verify(...), que les bons appels ont eu lieu — la vérification vit dans le mock, pas dans un simple assert.
Différence clé : avec un stub, l'assertion finale porte sur l'état et reste un assert classique ; avec un mock, la vérification finale porte sur les interactions et passe par Mockito.verify(...), qui interroge le mock lui-même.

(Référence : Fake and Mock Objects in Pictures, Bill Wake, https://www.industriallogic.com/blog/mock-objects-pictures/)

Stubs écrits à la main

Un stub implémente l'interface de la dépendance avec des réponses codées en dur. Aucune bibliothèque nécessaire.

// Interface de la dépendance
public interface Sport {
    String getNom();
    int getNbMoniteursNecessaires();
}

// Stub : réponses fixes, juste ce qu'il faut pour le test
public class SportStub implements Sport {

    @Override
    public String getNom() {
        return "natation";
    }

    @Override
    public int getNbMoniteursNecessaires() {
        return 2;
    }
}

Le test utilise le stub à la place de la vraie implémentation :

@Test
void moniteurPeutEncadrerSport() {
    Sport sport = new SportStub();
    Moniteur moniteur = new MoniteurImpl("Alice", sport);
    assertTrue(moniteur.getSports().contains(sport));
}
Le stub devient vite encombrant quand chaque test a besoin de réponses différentes : il faut alors un stub par variante. C'est exactement le problème que Mockito résout.

Imports utiles

import org.mockito.Mockito;

// Variante possible :
import static org.mockito.Mockito.*;

Imports pour l'utilisation par annotations avec JUnit Jupiter :

import org.junit.jupiter.api.extension.ExtendWith;

Convention pratique. Avec import static org.mockito.Mockito.*;, on écrit when(...) et verify(...) au lieu de Mockito.when(...) et Mockito.verify(...).

Mockito - bases

Classe principale : org.mockito.Mockito. Mockito permet de remplacer une dépendance par un objet simulé, appelé mock, dont on contrôle le comportement.

Créer un mock

static <T> T mock(Class<T> classToMock) static <T> T mock(Class<T> classToMock, String name) 
EmailGateway emailGateway = Mockito.mock(EmailGateway.class);

Un mock retourne des valeurs par défaut tant qu'aucun comportement n'est défini : false pour boolean, 0 pour les nombres, null pour les objets.

Définir le comportement d'un mock

static <T> OngoingStubbing<T> when(T methodCall) when(mock.methode(...)).thenReturn(valeur) when(mock.methode(...)).thenThrow(exception) 
Mockito.when(emailGateway.estDisponible()).thenReturn(true);
Mockito.when(emailGateway.envoyer("alice@example.com", "Bonjour")).thenReturn(true);

Plusieurs retours peuvent être chaînés pour des appels successifs :

Mockito.when(repository.count()).thenReturn(1, 2, 3);

Méthodes void

Pour une méthode void, on utilise la famille doXxx.

doNothing().when(mock).methodeVoid(...) doThrow(Throwable...).when(mock).methodeVoid(...) doReturn(Object).when(mock).methode(...) 
Mockito.doThrow(new IllegalStateException())
       .when(notificationService)
       .envoyer("alice@example.com");

Spy

static <T> T spy(T object) static <T> T spy(Class<T> classToSpy)

Un spy encapsule un vrai objet. Les méthodes réelles sont appelées par défaut, sauf celles explicitement stubées. C'est l'implémentation Mockito de la notion conceptuelle de spy : un stub qui enregistre en plus ses appels pour permettre une vérification a posteriori.

Utiliser les spies avec prudence. Un mock simple est souvent plus clair pour isoler la classe testée.

Matchers d'arguments

Classe : org.mockito.ArgumentMatchers. Mockito en hérite, donc on peut appeler ces méthodes via Mockito.anyString() ou avec un import statique.

Méthode Signification
any(), any(Class<T>) N'importe quelle valeur du type demandé.
anyString(), anyInt(), anyBoolean(), etc. N'importe quelle valeur du type primitif ou courant.
eq(T value) Valeur égale à celle donnée.
isNull(), notNull(), nullable(Class<T>) Correspondance sur la nullité.
contains(String), startsWith(String), endsWith(String), matches(String) Correspondances utiles pour les chaînes.
argThat(ArgumentMatcher<T>) Correspondance personnalisée par lambda ou matcher. 
Mockito.when(emailGateway.envoyer(Mockito.anyString(), Mockito.anyString()))
       .thenReturn(true);

Mockito.verify(emailGateway)
       .envoyer(Mockito.eq("alice@example.com"), Mockito.anyString());
Si un argument d'un appel Mockito utilise un matcher, les autres arguments de ce même appel doivent aussi utiliser des matchers. Exemple : utiliser eq("alice@example.com") avec anyString().

Mockito - vérifications

Après avoir exécuté la méthode testée, Mockito permet de vérifier que le mock a reçu ou non certains appels.

Vérification simple

static <T> T verify(T mock) static <T> T verify(T mock, VerificationMode mode) 
Mockito.verify(emailGateway).envoyer("alice@example.com", "Bonjour");

Nombre d'appels

Méthode Effet Exemple
times(int wantedNumberOfInvocations) Exactement ce nombre d'appels. verify(mock, times(1)).save(obj);
never() Aucun appel. Équivalent lisible de times(0). verify(mock, never()).delete(id);
atLeastOnce() Au moins un appel. verify(mock, atLeastOnce()).findAll();
atLeast(int minNumberOfInvocations) Au moins ce nombre d'appels. verify(mock, atLeast(2)).log(anyString());
atMost(int maxNumberOfInvocations) Au plus ce nombre d'appels. verify(mock, atMost(1)).connect();
only() Cet appel est le seul appel fait au mock. verify(mock, only()).save(obj); 
Mockito.verify(emailGateway, Mockito.times(1))
       .envoyer("alice@example.com", "Bonjour");

Mockito.verify(emailGateway, Mockito.never())
       .envoyer(Mockito.anyString(), Mockito.anyString());

Aucune interaction restante

static void verifyNoMoreInteractions(Object... mocks) static void verifyNoInteractions(Object... mocks)

verifyNoMoreInteractions vérifie qu'il ne reste pas d'appel non vérifié sur les mocks. verifyNoInteractions vérifie qu'aucun appel n'a eu lieu (remplace l'ancienne méthode dépréciée verifyZeroInteractions).

Ordre des appels

static InOrder inOrder(Object... mocks) <T> T InOrder.verify(T mock) <T> T InOrder.verify(T mock, VerificationMode mode) 
InOrder inOrder = Mockito.inOrder(repository);
inOrder.verify(repository).findById("42");
inOrder.verify(repository).save(objet);

Capturer un argument

Classe : org.mockito.ArgumentCaptor<T>.

static <U, S extends U> ArgumentCaptor<U> forClass(Class<S> clazz) T capture() T getValue() List<T> getAllValues() 
ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);

Mockito.verify(emailGateway).envoyer(captor.capture(), Mockito.eq("Bonjour"));

assertEquals("alice@example.com", captor.getValue());

Mockito - annotations avec JUnit Jupiter

Mockito peut être utilisé uniquement avec Mockito.mock(...). Il peut aussi être activé par annotations avec l'extension JUnit Jupiter.

MockitoExtension

Classe : org.mockito.junit.jupiter.MockitoExtension. Elle initialise les champs annotés par Mockito avant les tests. On l'enregistre avec l'annotation JUnit @ExtendWith.

@ExtendWith(MockitoExtension.class) 
import org.junit.jupiter.api.extension.ExtendWith;
import org.mockito.InjectMocks;
import org.mockito.Mock;
import org.mockito.junit.jupiter.MockitoExtension;

@ExtendWith(MockitoExtension.class)
class FacturationServiceTest {

    @Mock
    ClientRepository clientRepository;

    @Mock
    EmailGateway emailGateway;

    @InjectMocks
    FacturationService service;
}

@Mock

Crée automatiquement un mock pour le champ annoté.

@Mock @Mock(name = String)

@InjectMocks

Demande à Mockito de créer l'objet testé et d'y injecter les mocks disponibles, par constructeur, setter ou champ si possible.

@InjectMocks

@Spy

Crée un spy sur un vrai objet. Les méthodes réelles sont appelées par défaut.

@Spy

@Captor

Crée un ArgumentCaptor typé pour capturer les arguments passés aux mocks.

@Captor
Mockito.mock(...) est souvent plus explicite et ne demande pas de retenir les annotations Mockito. Les annotations sont utiles quand une classe de test contient plusieurs mocks.

Exemple complet neutre

Exemple complet combinant JUnit Jupiter, assertions et Mockito avec des classes fictives.

import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;
import org.mockito.Mockito;

import static org.junit.jupiter.api.Assertions.*;

class NotificationServiceTest {

    private EmailGateway emailGateway;
    private NotificationService notificationService;

    @BeforeEach
    void setUp() {
        emailGateway = Mockito.mock(EmailGateway.class);
        notificationService = new NotificationService(emailGateway);
    }

    @Test
    void notifierAvecAdresseVideLanceException() {
        assertThrows(IllegalArgumentException.class,
            () -> notificationService.notifier("", "Bonjour"));
    }

    @Test
    void notifierRetourneTrueQuandMessageEnvoye() {
        Mockito.when(emailGateway.estDisponible()).thenReturn(true);
        Mockito.when(emailGateway.envoyer("alice@example.com", "Bonjour")).thenReturn(true);

        assertTrue(notificationService.notifier("alice@example.com", "Bonjour"));

        Mockito.verify(emailGateway, Mockito.times(1))
               .envoyer("alice@example.com", "Bonjour");
    }

    @Test
    void notifierNeTentePasEnvoiSiPasserelleIndisponible() {
        Mockito.when(emailGateway.estDisponible()).thenReturn(false);

        assertFalse(notificationService.notifier("alice@example.com", "Bonjour"));

        Mockito.verify(emailGateway, Mockito.never())
               .envoyer(Mockito.anyString(), Mockito.anyString());
    }
}

Pièges fréquents

Problème Cause probable Correction
Le mock retourne false ou null Aucun comportement n'a été défini pour cet appel exact. Ajouter when(...).thenReturn(...) avec les bons arguments.
verify(...) échoue alors que la méthode semble appelée Arguments différents ou appel sur un autre objet que le mock. Vérifier que la classe testée reçoit bien le mock et utiliser des matchers si nécessaire.
Erreur Mockito avec matchers Mélange entre valeur brute et matcher dans le même appel. Remplacer les valeurs brutes par eq(...) quand un autre argument utilise any....
UnnecessaryStubbingException Un when(...) configuré n'est jamais utilisé par le test. Supprimer le stubbing inutile, ou le déplacer dans le test qui en a besoin.
On mocke la classe testée elle-même Confusion entre objet testé et dépendances. Instancier la vraie classe testée ; ne mocker que ses dépendances.
Un test dépend d'un autre test État partagé entre tests (mock configuré dans un test, réutilisé dans un autre). Recréer les mocks dans @BeforeEach. Chaque test doit pouvoir être exécuté seul.

Référence rapide

Mockito

import org.mockito.Mockito;
import org.mockito.ArgumentCaptor;

Service mock = Mockito.mock(Service.class);

Mockito.when(mock.method("x")).thenReturn("resultat");
Mockito.when(mock.method(Mockito.anyString())).thenReturn("resultat");
Mockito.when(mock.method("y")).thenThrow(new IllegalStateException());

assertEquals("resultat", mock.method("x"));

Mockito.verify(mock).method("x");
Mockito.verify(mock, Mockito.times(1)).method("x");
Mockito.verify(mock, Mockito.never()).method("y");

ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
Mockito.verify(mock).method(captor.capture());
assertEquals("x", captor.getValue());

Annotations

@ExtendWith(MockitoExtension.class)
class MonServiceTest {

    @Mock
    Dependance dependance;

    @InjectMocks
    MonService service;
}
