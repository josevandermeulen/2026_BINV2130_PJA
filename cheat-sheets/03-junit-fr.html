Contexte d'utilisation

Cette documentation couvre les dépendances de test suivantes : org.junit.jupiter:junit-jupiter:5.11.0 et org.junit.jupiter:junit-jupiter-params:5.11.0 via l'agrégat JUnit.

Pour écrire des tests unitaires, les étudiants ont surtout besoin de :

  • org.junit.jupiter.api.Test et des annotations de cycle de vie comme BeforeEach.
  • org.junit.jupiter.api.Assertions pour vérifier les résultats.
  • org.junit.jupiter.params.ParameterizedTest et les sources de paramètres si des tests paramétrés sont souhaités.
Attention aux imports JUnit 4. Cette documentation utilise JUnit 5, aussi appelé JUnit Jupiter. Ne pas importer org.junit.Test ni org.junit.Assert. Utiliser org.junit.jupiter.api.Test et org.junit.jupiter.api.Assertions.

Structure d'un test JUnit Jupiter

Une classe de test JUnit Jupiter peut être package-private. Les méthodes de test peuvent aussi être package-private. Elles ne doivent pas être private et retournent généralement void.

class MaClasseTest {

    @BeforeEach
    void setUp() {
        // Préparation avant chaque test.
    }

    @Test
    void nomDuCasDeTest() {
        // Arrange : préparer les objets et les données.
        // Act : appeler la méthode testée.
        // Assert : vérifier le résultat.
    }
}

Un test réussit si la méthode se termine sans lancer d'exception inattendue et sans assertion en échec. Un test échoue si une assertion échoue ou si une exception non attendue est lancée.

Annotations principales

Package principal : org.junit.jupiter.api.

@Test

annotation méthode

Marque une méthode comme méthode de test. JUnit l'exécute comme un cas de test indépendant.

@Test 
@Test
void additionneDeuxNombres() {
    assertEquals(5, calculatrice.additionner(2, 3));
}

Une méthode annotée @Test ne doit pas être private. Elle peut déclarer throws Exception si nécessaire.

@BeforeEach

annotation cycle de vie

Indique qu'une méthode doit être exécutée avant chaque méthode de test de la classe. Elle sert à recréer un état propre pour chaque test.

@BeforeEach 
private Service service;

@BeforeEach
void setUp() {
    service = new Service();
}
Si la fixture a besoin d'une date relative à « maintenant » (par exemple l'historique de prix d'un produit), calculer la date une seule fois dans une constante nommée plutôt que d'appeler LocalDate.now() dans chaque test : private static final LocalDate DATE_ANNEE_PASSEE = LocalDate.now().minusYears(1);. Cela évite des tests qui deviennent aléatoirement instables (flaky) si l'horloge tourne entre deux appels.

@AfterEach

annotation cycle de vie

Indique qu'une méthode doit être exécutée après chaque test. Elle est utile pour libérer des ressources ou vérifier un état final commun.

@AfterEach

@BeforeAll et @AfterAll

annotation cycle de vie

@BeforeAll exécute une méthode une seule fois avant tous les tests de la classe. @AfterAll exécute une méthode une seule fois après tous les tests.

@BeforeAll @AfterAll

Par défaut, ces méthodes doivent être static. Elles peuvent être non statiques si la classe de test est annotée avec @TestInstance(TestInstance.Lifecycle.PER_CLASS).

@DisplayName

annotation lisibilité

Définit un nom lisible pour une classe de test ou une méthode de test dans les rapports.

@DisplayName(String value) 
@Test
@DisplayName("Additionne deux nombres positifs")
void additionneDeuxNombresPositifs() {
    assertEquals(5, calculatrice.additionner(2, 3));
}

@Disabled

annotation désactivation

Désactive temporairement une classe ou une méthode de test. Le test est ignoré et n'est pas exécuté.

@Disabled @Disabled(String reason) 
@Test
@Disabled("Cas couvert plus tard")
void testNonExecute() {
    fail("Ce code ne sera pas exécuté");
}

@Nested

annotation organisation

Permet de regrouper des tests dans des classes internes non statiques. Les tests imbriqués peuvent partager le contexte de la classe externe.

@Nested

@RepeatedTest

annotation répétition

Répète le même test plusieurs fois.

@RepeatedTest(int value) @RepeatedTest(value = int, name = String) @RepeatedTest(value = int, failureThreshold = int) 
@RepeatedTest(3)
void genereToujoursUnIdentifiant() {
    assertNotNull(service.genererId());
}

@Timeout

annotation durée

Échoue si la méthode annotée dépasse la durée donnée.

@Timeout(long value) @Timeout(value = long, unit = TimeUnit.SECONDS) 
@Test
@Timeout(2)
void termineEnMoinsDeDeuxSecondes() {
    service.traitementCourt();
}

@TestInstance

annotation cycle de vie

Contrôle la création des instances de classe de test.

@TestInstance(TestInstance.Lifecycle.PER_METHOD) @TestInstance(TestInstance.Lifecycle.PER_CLASS)
  • PER_METHOD : comportement par défaut, nouvelle instance de test pour chaque méthode.
  • PER_CLASS : une seule instance pour toute la classe, ce qui permet des @BeforeAll non statiques.

Assertions

Classe : org.junit.jupiter.api.Assertions. Toutes les méthodes utiles sont statiques. Une assertion vérifie qu'une condition attendue est remplie ; si ce n'est pas le cas, elle échoue en lançant une erreur d'assertion JUnit et le test est marqué en échec.

Messages d'échec

La plupart des assertions existent avec ces variantes :

assertXxx(arguments) assertXxx(arguments, String message) assertXxx(arguments, Supplier<String> messageSupplier)

Le message est affiché uniquement si l'assertion échoue. Le Supplier<String> permet de construire un message tardivement.

Vrai ou faux

Méthode Effet Exemple
assertTrue(boolean condition) Réussit si la condition vaut true. assertTrue(panier.contient("livre"));
assertFalse(boolean condition) Réussit si la condition vaut false. assertFalse(panier.contient("stylo"));
assertTrue(BooleanSupplier condition) Évalue une lambda retournant un booléen. assertTrue(() -> service.estValide());

Égalité et différence

Ordre des paramètres : valeur attendue, puis valeur réelle.

assertEquals(expected, actual) assertEquals(expected, actual, message) assertNotEquals(unexpected, actual) assertEquals(double expected, double actual, double delta) assertEquals(float expected, float actual, float delta)

Les surcharges existent pour les objets et plusieurs types primitifs : byte, short, int, long, char, float et double. Pour un booléen, on utilise le plus souvent assertTrue ou assertFalse; assertEquals peut aussi comparer des objets Boolean.

assertEquals(3, liste.size());
assertEquals("alice@example.com", utilisateur.getEmail());
assertNotEquals("admin", utilisateur.getRole());
assertEquals(12.5, prix, 0.001);
Pour les nombres à virgule flottante, utiliser une marge d'erreur avec delta. Exemple : assertEquals(0.3, resultat, 0.0001).

Nullité

assertNull(Object actual) assertNotNull(Object actual) 
assertNull(repository.findById("inconnu"));
assertNotNull(service.creerCommande("client-1"));

Identité d'objet

assertSame(Object expected, Object actual) assertNotSame(Object unexpected, Object actual)

assertSame vérifie que les deux références pointent vers le même objet, pas seulement vers deux objets égaux.

Tester equals et hashCode

Quand une classe métier redéfinit equals, un test dédié vérifie que deux instances distinctes mais avec le même contenu sont bien considérées égales, qu'un attribut différent suffit à rendre les objets inégaux, et que hashCode reste cohérent avec equals (deux objets égaux doivent avoir le même hashCode).

@Test
@DisplayName("Deux articles avec le même nom et la même catégorie sont égaux")
void testEquals() {
    Article article = new Article("Cahier", "Papeterie");
    assertEquals(articleReference, article);    // même contenu, référence différente
    assertNotSame(articleReference, article);   // ce n'est pas le même objet
}

@Test
@DisplayName("Un nom différent rend les articles inégaux")
void testNonEquals() {
    Article article = new Article("Stylo", "Papeterie");
    assertNotEquals(articleReference, article);
}

@Test
@DisplayName("Deux articles égaux ont le même hashCode")
void testHashCode() {
    Article article = new Article("Cahier", "Papeterie");
    assertEquals(articleReference.hashCode(), article.hashCode());
}
Le test d'égalité doit comparer à un objet différent en référence mais égal en contenu (pas articleReference lui-même) : sinon assertEquals réussirait même si equals n'avait jamais été redéfini, à cause de l'égalité par défaut héritée d'Object qui compare les références. assertNotSame confirme que ce n'est bien pas le même objet. Le test de hashCode vérifie la cohérence avec equals, pas une valeur de hash précise.

Tableaux, itérables et lignes

assertArrayEquals(expectedArray, actualArray) assertArrayEquals(double[] expected, double[] actual, double delta) assertIterableEquals(Iterable<?> expected, Iterable<?> actual) assertLinesMatch(List<String> expectedLines, List<String> actualLines)

assertArrayEquals compare le contenu des tableaux. assertIterableEquals compare l'ordre et le contenu d'itérables comme des listes.

Regrouper plusieurs assertions

assertAll(Executable... executables) assertAll(String heading, Executable... executables)

assertAll exécute toutes les assertions fournies. Si plusieurs échouent, JUnit les regroupe dans un seul rapport d'échec.

assertAll("commande valide",
    () -> assertEquals("CMD-1", commande.getNumero()),
    () -> assertEquals(3, commande.getArticles().size()),
    () -> assertTrue(commande.estPayee())
);

Vérifier une exception

<T extends Throwable> T assertThrows(Class<T> expectedType, Executable executable) <T extends Throwable> T assertThrowsExactly(Class<T> expectedType, Executable executable)

assertThrows réussit si l'exécutable lance une exception du type attendu ou d'un sous-type. assertThrowsExactly exige exactement le type indiqué.

IllegalArgumentException ex = assertThrows(
    IllegalArgumentException.class,
    () -> compte.retirer(-10)
);

assertEquals("Le montant doit être positif", ex.getMessage());

assertThrows fonctionne aussi bien avec les exceptions du JDK qu'avec une exception métier définie dans le projet (par exemple une exception custom qui étend Exception ou RuntimeException) :

@Test
void testAjouterPrixDateDejaPresente() {
    assertThrows(DateDejaPresenteException.class,
        () -> produit.ajouterPrix(dateDejaUtilisee, nouveauPrix));
}

Vérifier l'absence d'exception

assertDoesNotThrow(Executable executable) <T> T assertDoesNotThrow(ThrowingSupplier<T> supplier) 
assertDoesNotThrow(() -> compte.deposer(50));

int resultat = assertDoesNotThrow(() -> calculatrice.diviser(10, 2));

Échouer explicitement

fail() fail(String message) fail(Throwable cause)

fail marque immédiatement le test comme échoué. C'est utile dans une branche qui ne devrait jamais être atteinte.

Durée d'exécution

assertTimeout(Duration timeout, Executable executable) assertTimeoutPreemptively(Duration timeout, Executable executable)

assertTimeout attend la fin du code puis échoue si la durée est dépassée. assertTimeoutPreemptively interrompt l'exécution quand le délai expire, ce qui peut avoir des effets de bord avec des threads ou des ressources partagées.

Type d'objet

<T> T assertInstanceOf(Class<T> expectedType, Object actual)

Vérifie qu'un objet est une instance d'un type donné et retourne l'objet typé.

Object resultat = service.getResultat();
String texte = assertInstanceOf(String.class, resultat);

Exceptions et lambdas

Type fonctionnel : org.junit.jupiter.api.function.Executable.

void execute() throws Throwable

JUnit utilise Executable pour représenter un bloc de code susceptible de lancer une exception. En pratique, on écrit presque toujours une lambda.

// Correct : le code est exécuté par assertThrows.
assertThrows(IllegalArgumentException.class,
    () -> service.creerUtilisateur(null)
);

// Incorrect : la méthode est appelée avant assertThrows.
assertThrows(IllegalArgumentException.class,
    service.creerUtilisateur(null)
);
Si la méthode testée prend des paramètres, l'expression lambda doit contenir l'appel complet : () -> objet.methode(parametre).

Tests paramétrés

Package principal : org.junit.jupiter.params et org.junit.jupiter.params.provider. Un test paramétré s'écrit avec @ParameterizedTest, pas avec @Test.

@ParameterizedTest

annotation méthode

Marque une méthode comme test exécuté plusieurs fois avec des paramètres différents.

@ParameterizedTest @ParameterizedTest(name = String) 
@ParameterizedTest
@ValueSource(strings = {"alice", "bob"})
void nomNonVide(String nom) {
    assertFalse(nom.isBlank());
}

@ValueSource

source

Fournit une liste simple de valeurs d'un seul type.

@ValueSource(strings = {...}) @ValueSource(ints = {...}) @ValueSource(longs = {...}) @ValueSource(doubles = {...}) @ValueSource(booleans = {...}) @ValueSource(classes = {...})

Autres attributs disponibles : shorts, bytes, floats, chars.

@CsvSource

source

Fournit plusieurs colonnes de valeurs sous forme de chaînes CSV intégrées dans l'annotation.

@CsvSource({"a, 1", "b, 2"}) @CsvSource(value = {...}, delimiter = ';') @CsvSource(value = {...}, nullValues = {"NULL"}) 
@ParameterizedTest
@CsvSource({
    "2, 3, 5",
    "4, 5, 9"
})
void addition(int a, int b, int attendu) {
    assertEquals(attendu, calculatrice.additionner(a, b));
}

Attributs utiles : delimiter, delimiterString, quoteCharacter, emptyValue, nullValues, ignoreLeadingAndTrailingWhitespace.

@MethodSource

source

Fournit les paramètres à partir d'une méthode. La méthode source retourne par exemple un Stream, une Collection, un tableau ou un Stream<Arguments>.

@MethodSource @MethodSource("nomDeMethode") 
static Stream<Arguments> casAdditions() {
    return Stream.of(
        Arguments.of(2, 3, 5),
        Arguments.of(4, 5, 9)
    );
}

@ParameterizedTest
@MethodSource("casAdditions")
void additionAvecMethodSource(int a, int b, int attendu) {
    assertEquals(attendu, calculatrice.additionner(a, b));
}

Si la classe de test utilise le cycle de vie par défaut, la méthode source est généralement static.

Arguments

interface

Représente un ensemble d'arguments pour une invocation de test paramétré.

static Arguments of(Object... arguments) static Arguments arguments(Object... arguments)

@NullSource, @EmptySource, @NullAndEmptySource

source

@NullSource fournit null. @EmptySource fournit une valeur vide adaptée au type, par exemple "" pour String. @NullAndEmptySource combine les deux.

@ParameterizedTest
@NullAndEmptySource
void nomUtilisateurInvalide(String nom) {
    assertThrows(IllegalArgumentException.class,
        () -> utilisateurService.creerUtilisateur(nom));
}

@EnumSource

source

Fournit les constantes d'un enum. Les attributs names et mode permettent de filtrer les constantes.

@EnumSource(MonEnum.class) @EnumSource(value = MonEnum.class, names = {"A", "B"})

@CsvFileSource

source

Lit des paramètres depuis un fichier CSV dans les ressources disponibles ou depuis un chemin de fichier.

@CsvFileSource(resources = "/cas.csv") @CsvFileSource(files = "src/test/resources/cas.csv", numLinesToSkip = 1)

Exemple complet neutre

Exemple complet combinant annotations, assertions et test paramétré avec des classes fictives.

import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.ValueSource;

import static org.junit.jupiter.api.Assertions.*;

class CompteBancaireTest {

    private CompteBancaire compte;

    @BeforeEach
    void setUp() {
        compte = new CompteBancaire("BE-001");
        compte.deposer(100);
    }

    @Test
    @DisplayName("Un dépôt augmente le solde")
    void deposerAugmenteLeSolde() {
        compte.deposer(50);
        assertEquals(150, compte.getSolde());
    }

    @Test
    @DisplayName("Un retrait supérieur au solde lance une exception")
    void retirerPlusQueLeSoldeLanceException() {
        assertThrows(IllegalStateException.class,
            () -> compte.retirer(500));
    }

    @ParameterizedTest
    @ValueSource(ints = {0, -1, -50})
    @DisplayName("Un dépôt non strictement positif est refusé")
    void deposerMontantInvalideLanceException(int montant) {
        assertThrows(IllegalArgumentException.class,
            () -> compte.deposer(montant));
    }

    @Test
    @DisplayName("État complet du compte après création")
    void etatInitialComplet() {
        CompteBancaire nouveau = new CompteBancaire("BE-002");
        assertAll("nouveau compte",
            () -> assertEquals("BE-002", nouveau.getNumero()),
            () -> assertEquals(0, nouveau.getSolde()),
            () -> assertFalse(nouveau.estBloque())
        );
    }
}

Développer des tests avec l'IA

Un assistant IA (Claude, ChatGPT, Copilot, …) peut faire gagner du temps sur les tests : proposer des cas limites auxquels on n'a pas pensé, générer la répétition mécanique d'un test paramétré, ou reformuler un @DisplayName plus clair. C'est un bon usage : l'IA complète la réflexion, elle ne la remplace pas.

Ce qui ne marche pas. Demander à l'IA d'écrire les tests sans avoir compris soi-même ce que la méthode testée est censée faire. L'IA n'a pas accès au comportement métier voulu par l'énoncé — seulement au code qu'on lui montre — et peut donc :
  • halluciner une méthode ou un attribut qui n'existe pas dans la classe ;
  • se tromper sur une règle métier si elle n'est pas explicitement donnée (par exemple un plancher ou un plafond non mentionné dans le code) ;
  • écrire un test qui passe sans vraiment vérifier le bon comportement — un test qui recopie l'implémentation au lieu de vérifier le résultat attendu ne détecte aucune régression.
Réflexe de vérification. Un test généré par IA doit être relu et compris ligne par ligne avant d'être gardé, exactement comme un test écrit à la main. Le meilleur moyen de vérifier qu'un test teste vraiment quelque chose : casser volontairement la méthode testée (inverser une condition, changer une valeur) et vérifier que le test passe au rouge. S'il reste vert alors que le code est cassé, le test ne sert à rien, qu'il vienne d'une IA ou non.

Pièges spécifiques à equals/hashCode

Le contrat equals/hashCode a deux moitiés : un test généré par IA peut ne couvrir que l'égalité et oublier hashCode, ou l'inverse. Vérifier que les deux sont bien testées séparément (voir Tester equals et hashCode).

La distinction égalité/identité est également subtile : un test généré par IA peut réutiliser la même référence là où le scénario demande justement un objet égal (equals) mais de référence différente (assertNotSame) — ce test-là ne vérifierait pas la bonne chose, puisqu'il resterait vert même si equals n'était jamais appelé.

Pièges fréquents

Problème Cause probable Correction
@Test non reconnu Mauvais import JUnit 4. Importer org.junit.jupiter.api.Test.
assertThrows ne compile pas L'appel testé n'est pas dans une lambda. Écrire () -> objet.methode().
Un test dépend d'un autre test État partagé entre tests. Réinitialiser l'état dans @BeforeEach. Chaque test doit pouvoir être exécuté seul.
@ParameterizedTest ne s'exécute pas Pas de source de paramètres. Ajouter @ValueSource, @CsvSource, @MethodSource, etc.
assertEquals échoue sur deux objets « identiques » La classe ne redéfinit pas equals. Redéfinir equals/hashCode, ou comparer les attributs un à un.
Comparaison de double échoue de peu Imprécision des nombres à virgule flottante. Utiliser la surcharge avec delta : assertEquals(expected, actual, 0.0001).
