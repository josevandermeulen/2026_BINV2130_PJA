Programmer par interfaces

Le code client dépend d'interfaces, jamais des classes concrètes. Les implémentations (QueryImpl, ServiceImpl…) restent invisibles hors de leur package (visibilité package-private).

// Visible partout : le contrat
public interface Query {
    String getUrl();
    String execute();
}

// Package-private : personne ne peut faire new QueryImpl() hors du package
class QueryImpl implements Query {
    ...
}

Avantages : on peut changer l'implémentation sans toucher au code client, et la remplacer par un mock dans les tests.

Question immédiate : si le client ne peut pas faire new QueryImpl(), qui crée les objets ? Réponse : la factory.

Le pattern Factory

Factory method

Une méthode dédiée centralise la création : le client demande un objet du type interface, la factory choisit la classe concrète.

public interface QueryFactory {
    Query getQuery();
}

class QueryFactoryImpl implements QueryFactory {
    @Override
    public Query getQuery() {
        return new QueryImpl();   // seule la factory connaît QueryImpl
    }
}

Abstract Factory

Une interface de factory dont plusieurs implémentations produisent des familles d'objets différentes (par exemple une factory de production et une factory de test). Le client reçoit la factory via son constructeur et ne sait pas laquelle il utilise.

Factory configurable

Le choix de l'implémentation peut venir d'une configuration (fichier, propriété système) plutôt que d'être codé en dur — souvent combiné avec l'introspection (Class.forName(...).getConstructor().newInstance()) pour instancier la classe dont le nom est lu dans la configuration.

Injection de dépendances (DI)

Un objet ne crée pas lui-même ses dépendances : elles lui sont fournies de l'extérieur, idéalement par le constructeur. C'est l'inverse du new interne.

// Sans DI : couplage fort, non testable
public class ProxyServer {
    private QueryFactory factory = new QueryFactoryImpl();   // figé
}

// Avec DI par constructeur : découplé, testable
public class ProxyServer {

    private final QueryFactory factory;

    public ProxyServer(QueryFactory factory) {   // la dépendance est injectée
        this.factory = factory;
    }
}

La composition de l'application (qui reçoit quoi) se fait en un seul endroit, typiquement le main :

public static void main(String[] args) {
    QueryFactory factory = new QueryFactoryImpl();
    ProxyServer serveur = new ProxyServer(factory);   // injection
    serveur.demarrer();
}
DI + interfaces + factory = chaque pièce est remplaçable : une autre implémentation en production, un mock dans les tests (voir le cheat sheet semaine 5).

Fichiers .properties - configuration sans recompilation

Paires clé/valeur dans un fichier texte, lu au démarrage par java.util.Properties. Le fichier vit à la racine du projet (puis à côté du JAR), jamais dans src : il doit rester modifiable sans recompiler.

# blacklist.properties
blacklistedDomains=google.be;facebook.com;instagram.com
void load(InputStream inStream) String getProperty(String key) String getProperty(String key, String defaultValue) 
Properties properties = new Properties();
try (FileInputStream input = new FileInputStream("blacklist.properties")) {
    properties.load(input);
} catch (IOException e) {
    throw new RuntimeException("Fichier de configuration introuvable", e);
}
Set<String> domaines = Set.of(properties.getProperty("blacklistedDomains", "").split(";"));
Charger le fichier une seule fois (bloc/champ static ou constructeur), pas à chaque requête.

Injection par setter

Variante de la DI : la dépendance est injectée après l'instanciation, via une méthode publique, au lieu du constructeur.

public class ProxyServer {

    private QueryFactory factory;              // plus final

    public void setFactory(QueryFactory factory) {
        this.factory = factory;
    }
}

// Composition dans le main
ProxyServer serveur = new ProxyServer();
serveur.setFactory(new QueryFactoryImpl());   // injection
serveur.demarrer();

Setter : pour une dépendance optionnelle ou remplaçable après création. Constructeur : pour une dépendance obligatoire et définitive (le champ peut rester final).

Injection automatique (@Inject + réflexion)

Principe des conteneurs IoC (Spring…) : on annote les champs à injecter, un injecteur les remplit par réflexion (voir cheat sheet semaine 11). Plus aucun assemblage manuel dans le main.

@Retention(RetentionPolicy.RUNTIME)   // lisible à l'exécution
@Target(ElementType.FIELD)            // uniquement sur des champs
public @interface Inject {
    Class<?> value();                 // l'implémentation à instancier
}

public class ProxyServer {
    @Inject(QueryFactoryImpl.class)
    private QueryFactory queryFactory;
}

public class Injector {
    public static void inject(Object target) {
        for (Field field : target.getClass().getDeclaredFields()) {
            if (!field.isAnnotationPresent(Inject.class)) {
                continue;
            }
            Class<?> impl = field.getAnnotation(Inject.class).value();
            field.setAccessible(true);            // champ privé
            try {
                field.set(target, impl.getDeclaredConstructor().newInstance());
            } catch (ReflectiveOperationException e) {
                throw new RuntimeException("Injection impossible : " + field.getName(), e);
            }
        }
    }
}

// main : l'injecteur assemble tout
ProxyServer serveur = new ProxyServer();
Injector.inject(serveur);
serveur.demarrer();
Sans @Retention(RetentionPolicy.RUNTIME), l'annotation disparaît à la compilation : isAnnotationPresent renvoie toujours false.

Javadoc & packaging JAR

Javadoc

Commentaires /** ... */ sur les interfaces publiques et leurs méthodes ; tags @param, @return, @throws, @deprecated. Génération : IntelliJ → Tools → Generate JavaDoc (output : dossier doc). Les classes package-private (ex. QueryImpl) n'apparaissent pas dans la doc générée — c'est l'encapsulation ; la javadoc des implémentations publiques est héritée des interfaces.

JAR exécutable avec Maven

<plugin>
    <artifactId>maven-assembly-plugin</artifactId>
    <configuration>
        <archive>
            <manifest>
                <mainClass>main.Main</mainClass>
            </manifest>
        </archive>
        <descriptorRefs>
            <descriptorRef>jar-with-dependencies</descriptorRef>
        </descriptorRefs>
    </configuration>
</plugin>
mvn clean compile assembly:single   # JAR dans target/
java -jar target/monapp-jar-with-dependencies.jar
Les fichiers .properties ne sont pas dans le JAR : ils doivent se trouver dans le dossier depuis lequel on lance java -jar.

HttpClient - envoyer une requête HTTP

Package java.net.http (Java 11+). Trois classes : HttpClient, HttpRequest, HttpResponse.

static HttpClient newHttpClient() <T> HttpResponse<T> send(HttpRequest request, HttpResponse.BodyHandler<T> handler) <T> CompletableFuture<HttpResponse<T>> sendAsync(HttpRequest request, HttpResponse.BodyHandler<T> handler) 
HttpClient client = HttpClient.newHttpClient();

HttpRequest requete = HttpRequest.newBuilder()
    .uri(URI.create("https://example.org/data"))
    .GET()
    .build();

// Synchrone
HttpResponse<String> reponse =
    client.send(requete, HttpResponse.BodyHandlers.ofString());
System.out.println(reponse.statusCode());
System.out.println(reponse.body());

// Asynchrone : renvoie un CompletableFuture<HttpResponse<String>>
client.sendAsync(requete, HttpResponse.BodyHandlers.ofString());

La mécanique de CompletableFuture (thenApply, thenAccept, join…) est couverte en détail dans le cheat sheet semaine 9.

Scanner - lecture au clavier

Classe : java.util.Scanner, branchée sur System.in.

Scanner(InputStream source) String nextLine() int nextInt() boolean hasNextLine() 
Scanner scanner = new Scanner(System.in);
System.out.print("URL à interroger : ");
String url = scanner.nextLine();
Après un nextInt(), le retour à la ligne reste dans le tampon : un nextLine() qui suit renvoie une chaîne vide. Consommer le reste de ligne avec un nextLine() intermédiaire.

Exemple complet neutre

Mini-application combinant interface, factory, DI et requête HTTP asynchrone :

public interface Telechargeur {
    CompletableFuture<String> telecharger(String url);
}

class TelechargeurHttp implements Telechargeur {

    private final HttpClient client = HttpClient.newHttpClient();

    @Override
    public CompletableFuture<String> telecharger(String url) {
        HttpRequest requete = HttpRequest.newBuilder()
            .uri(URI.create(url)).GET().build();
        return client.sendAsync(requete, HttpResponse.BodyHandlers.ofString())
                     .thenApply(HttpResponse::body);
    }
}

public interface TelechargeurFactory {
    Telechargeur creer();
}

class TelechargeurFactoryImpl implements TelechargeurFactory {
    @Override
    public Telechargeur creer() {
        return new TelechargeurHttp();
    }
}

public class Application {

    private final Telechargeur telechargeur;

    public Application(Telechargeur telechargeur) {   // DI par constructeur
        this.telechargeur = telechargeur;
    }

    public void afficher(String url) {
        telechargeur.telecharger(url)
                    .thenAccept(System.out::println)
                    .join();
    }

    public static void main(String[] args) {
        TelechargeurFactory factory = new TelechargeurFactoryImpl();
        Application app = new Application(factory.creer());
        app.afficher("https://example.org");
    }
}

Pièges fréquents

Problème Cause probable Correction
new QueryImpl() ne compile pas depuis le client La classe est package-private, c'est voulu. Passer par la factory du package.
Impossible de tester une classe isolément Elle fait new sur ses dépendances en interne. Injecter les dépendances par le constructeur ; mocker dans les tests.
Le client HTTP async ne produit rien Le main se termine avant la fin du CompletableFuture. join() sur le future final (voir semaine 9).
nextLine() renvoie une chaîne vide Retour à la ligne resté dans le tampon après nextInt(). Appeler un nextLine() de purge avant.
Types concrets qui fuient dans les signatures Retour QueryImpl au lieu de Query. Toujours déclarer les types interface dans les signatures publiques.
isAnnotationPresent renvoie toujours false Annotation sans rétention runtime. Ajouter @Retention(RetentionPolicy.RUNTIME).
IllegalAccessException en injectant un champ privé field.set sur un champ non accessible. field.setAccessible(true) avant field.set.
Le JAR ne trouve pas le fichier .properties Le fichier n'est pas embarqué dans le JAR (et ne doit pas l'être). Le copier à côté du JAR, dans le dossier de lancement.
