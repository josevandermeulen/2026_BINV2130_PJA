Notions essentielles

L'introspection, aussi appelée réflexion, permet d'obtenir des informations sur un programme Java pendant son exécution. Elle permet par exemple de connaître les méthodes déclarées par une classe, de lire les annotations présentes sur une méthode, de créer un objet à partir d'un Class<?>, ou d'appeler dynamiquement une méthode.

Les classes principales se trouvent dans les packages java.lang, java.lang.reflect et java.lang.annotation.

Type Rôle
Class<?> Représente une classe, une interface, un enum, une annotation ou un type primitif à l'exécution.
Constructor<T> Représente un constructeur et permet de créer une instance.
Method Représente une méthode et permet de l'inspecter ou de l'appeler.
Field Représente un attribut et permet de l'inspecter ou d'en lire/modifier la valeur.
Annotation Représente une annotation lue à l'exécution si sa rétention est RUNTIME.
La réflexion est puissante mais contourne une partie des vérifications habituelles du compilateur. Il faut gérer explicitement les exceptions et vérifier les types avant d'invoquer dynamiquement du code.

Class<?>

La classe java.lang.Class représente le type d'un objet ou d'une déclaration Java à l'exécution.

Obtenir un objet Class

MaClasse.class objet.getClass() Class.forName(String className) 
Class<?> type1 = Produit.class;
Class<?> type2 = produit.getClass();
Class<?> type3 = Class.forName("demo.Produit");

.class est vérifié à la compilation. getClass() part d'un objet existant. Class.forName charge une classe depuis son nom complet et peut lancer ClassNotFoundException.

Informations générales

String getName() String getSimpleName() String getPackageName() Class<? super T> getSuperclass() Class<?>[] getInterfaces() boolean isInterface() boolean isAnnotation() boolean isEnum() boolean isArray() boolean isPrimitive() 
Class<?> type = Produit.class;

System.out.println(type.getName());       // demo.Produit
System.out.println(type.getSimpleName()); // Produit
System.out.println(type.isInterface());   // false

Membres publics ou déclarés

Les méthodes commençant par get retournent généralement les membres publics accessibles, y compris ceux hérités. Les méthodes commençant par getDeclared retournent les membres déclarés directement dans la classe, y compris les membres privés, mais pas les membres hérités.

Method[] getMethods() Method[] getDeclaredMethods() Field[] getFields() Field[] getDeclaredFields() Constructor<?>[] getConstructors() Constructor<?>[] getDeclaredConstructors()
Méthode Ce qui est retourné
getMethods() Méthodes publiques de la classe, des superclasses et interfaces.
getDeclaredMethods() Méthodes déclarées directement dans la classe, quelle que soit leur visibilité.
getFields() Champs publics accessibles, y compris hérités.
getDeclaredFields() Champs déclarés directement dans la classe, quelle que soit leur visibilité.
L'ordre des éléments retournés par la réflexion n'est pas une garantie portable. Ne pas écrire une logique qui dépend de l'ordre exact des méthodes ou des champs.

Appeler une méthode

Object invoke(Object obj, Object... args)

invoke appelle la méthode représentée par l'objet Method. Pour une méthode d'instance, le premier argument est l'objet cible. Pour une méthode statique, on peut passer null comme objet cible.

Produit produit = new Produit("Stylo", 2.5);

Method getNom = Produit.class.getDeclaredMethod("getNom");
String nom = (String) getNom.invoke(produit);

Method changerPrix = Produit.class.getDeclaredMethod("changerPrix", double.class);
changerPrix.invoke(produit, 3.0);

La valeur de retour est de type Object. Si la méthode appelée retourne void, invoke retourne null.

Méthodes privées ou non publiques

void setAccessible(boolean flag)

setAccessible(true) demande à Java d'autoriser l'accès à un membre non public. Cette opération doit être utilisée avec prudence.

Method methode = Produit.class.getDeclaredMethod("calculerMarge");
methode.setAccessible(true);
double marge = (double) methode.invoke(produit);

Annotations

Une annotation personnalisée se déclare avec @interface. Pour qu'elle soit lisible par introspection pendant l'exécution, sa rétention doit être RetentionPolicy.RUNTIME.

Déclarer une annotation

@Retention(RetentionPolicy.RUNTIME) @Target(ElementType.METHOD) public @interface MonAnnotation { ... } 
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface Action {
    String nom();
    int priorite() default 0;
}

Les méthodes déclarées dans une annotation sont ses attributs. Un attribut peut avoir une valeur par défaut avec default.

Cibles possibles

Enum : java.lang.annotation.ElementType.

Constante Cible
TYPE Classe, interface, enum ou annotation.
METHOD Méthode.
FIELD Attribut.
CONSTRUCTOR Constructeur.
PARAMETER Paramètre de méthode ou de constructeur.

Rétentions possibles

Enum : java.lang.annotation.RetentionPolicy.

Constante Effet
SOURCE L'annotation existe dans le code source seulement. Elle disparaît à la compilation.
CLASS L'annotation est présente dans le fichier .class, mais n'est pas disponible par réflexion à l'exécution.
RUNTIME L'annotation est disponible à l'exécution via la réflexion.

Lire une annotation

boolean isAnnotationPresent(Class<? extends Annotation> annotationClass) <A extends Annotation> A getAnnotation(Class<A> annotationClass) <A extends Annotation> A getDeclaredAnnotation(Class<A> annotationClass) Annotation[] getAnnotations() Annotation[] getDeclaredAnnotations() 
for (Method methode : Controleur.class.getDeclaredMethods()) {
    if (methode.isAnnotationPresent(Action.class)) {
        Action action = methode.getAnnotation(Action.class);
        System.out.println(action.nom());
        System.out.println(action.priorite());
    }
}

Ces méthodes existent sur Class, Method, Field et Constructor, car ces types héritent de l'interface AnnotatedElement.

Si getAnnotation(...) retourne null alors que l'annotation semble présente dans le code, vérifier que l'annotation est déclarée avec @Retention(RetentionPolicy.RUNTIME).

Annotations du mini-JUnit

Un mini-JUnit reconstruit le mécanisme de JUnit avec la réflexion : une annotation @MonTest (équivalent maison de @Test, déclarée exactement comme Action plus haut) marque les méthodes de test, et un TestRunner les découvre avec getDeclaredMethods() et isAnnotationPresent. Quatre annotations supplémentaires enrichissent le runner avec des fonctionnalités connues du vrai JUnit ; chacune ne demande rien de neuf sur le plan des API, seulement une combinaison différente de ce qui précède.

Annotation maison Équivalent JUnit Rôle
@AvantChaqueTest @BeforeEach Méthode de préparation invoquée avant chaque test, sur l'instance fraîchement créée.
@Affichage(String valeur) @DisplayName Libellé lisible affiché dans le rapport à la place du nom de la méthode.
@Repeter(int fois) @RepeatedTest Invoque la méthode fois fois, une nouvelle instance à chaque répétition.
@TestDesactive @Disabled La méthode annotée n'est jamais invoquée ; elle apparaît dans le rapport avec un statut « ignorée ».

@AvantChaqueTest : préparer chaque test

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface AvantChaqueTest {
}

Avant d'invoquer chaque méthode de test sur une instance, le TestRunner invoque d'abord toutes les méthodes de la classe annotées @AvantChaqueTest, sur cette même instance :

for (Method preparation : classe.getDeclaredMethods()) {
    if (preparation.isAnnotationPresent(AvantChaqueTest.class)) {
        preparation.invoke(instance);
    }
}

@Affichage : un libellé dans le rapport

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface Affichage {
    String valeur();
}

C'est une annotation à élément : le TestRunner lit sa valeur avec getAnnotation(Affichage.class).valeur() pour choisir le libellé à afficher, au lieu du nom brut de la méthode :

String libelle = methode.getName();
if (methode.isAnnotationPresent(Affichage.class)) {
    libelle = methode.getAnnotation(Affichage.class).valeur();
}

@Repeter : répéter un test

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface Repeter {
    int fois();
}

Le TestRunner lit l'élément fois et boucle : chaque répétition reçoit une nouvelle instance de la classe de test, pour que les répétitions restent isolées les unes des autres.

int fois = methode.isAnnotationPresent(Repeter.class)
    ? methode.getAnnotation(Repeter.class).fois()
    : 1;

for (int i = 0; i < fois; i++) {
    Object instance = classe.getDeclaredConstructor().newInstance(); // nouvelle instance à chaque répétition
    methode.invoke(instance);
}

@TestDesactive : ignorer un test

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface TestDesactive {
}

Une méthode annotée @TestDesactive n'est jamais passée à invoke : le TestRunner la détecte et ajoute directement un résultat de statut « ignorée » au rapport, sans exécuter le corps de la méthode.

if (methode.isAnnotationPresent(TestDesactive.class)) {
    rapport.ajouter(new ResultatTest(methode.getName(), Statut.IGNORE, null));
    continue; // jamais invoquée
}

Réflexion sur les champs (validateur par annotations)

Le mini-JUnit réfléchit sur les méthodes. Un mini-validateur d'objets — le miroir déclaratif du pattern Util.checkObject/checkString utilisé pour valider des paramètres à la main — applique le même mécanisme aux champs : découvrir les champs annotés d'un objet et lire leur valeur. Les API de base (getDeclaredFields(), Field.getName(), Field.get(instance)) sont déjà présentées à la section Field ci-dessus ; seul le contexte d'utilisation change ici.

Trouver les champs annotés d'une classe

getDeclaredFields() est le miroir de getDeclaredMethods() pour les attributs. Un validateur cherche, pour une annotation donnée, tous les champs qui la portent :

List<Field> trouverChampsAnnotes(Class<?> classe, Class<? extends Annotation> annotation) {
    List<Field> champs = new ArrayList<>();
    for (Field champ : classe.getDeclaredFields()) {
        if (champ.isAnnotationPresent(annotation)) {
            champs.add(champ);
        }
    }
    champs.sort(Comparator.comparing(Field::getName));
    return champs;
}

Comme pour getDeclaredMethods(), l'ordre de getDeclaredFields() n'est pas garanti : trier par nom rend le rapport reproductible.

setAccessible(true) et les limites de l'encapsulation

Les champs qu'un validateur doit lire sont presque toujours private — c'est même tout l'intérêt de l'encapsulation. Par défaut, la réflexion respecte les modificateurs d'accès : appeler champ.get(objet) sur un champ privé lance une IllegalAccessException. setAccessible(true) désactive ce contrôle avant la lecture.

void setAccessible(boolean flag) 
Field champ = classe.getDeclaredField("nom");
champ.setAccessible(true);                 // sans cette ligne : IllegalAccessException
Object valeur = champ.get(objet);

Pour un champ de type primitif (int, double...), champ.get(objet) renvoie la valeur emballée dans sa classe enveloppante (Integer, Double...), qui hérite de Number — utile pour valider un champ numérique quel que soit son type primitif exact.

private protège contre un accès accidentel depuis un autre code Java compilé normalement, pas contre la réflexion. N'importe quel code qui obtient un Field et appelle setAccessible(true) peut lire, et même modifier, un champ privé. L'encapsulation reste une bonne pratique de conception, mais ce n'est pas une barrière de sécurité face à un code qui a le droit d'utiliser la réflexion.
Déclarer une annotation ciblant un champ est identique à la partie méthode, à un détail près : @Target(ElementType.METHOD) devient @Target(ElementType.FIELD) (voir le tableau des cibles à la section Annotations). 
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.FIELD)
public @interface NonNul {
}

Validation déclarative par annotations

Une validation écrite « à la main » place la règle dans le constructeur, sous forme d'appel impératif (Util.checkObject(nom), Util.checkString(matricule)...) répété de classe en classe. Une validation déclarative inverse l'approche : la règle est posée une seule fois, comme annotation sur le champ, et un mécanisme générique — un validateur, utilisant la réflexion sur les champs — la fait respecter pour n'importe quelle classe.

Les annotations de contrainte

Annotation Contrainte exprimée
@NonNul Le champ ne peut pas être null.
@LongueurMin(int valeur) Une String doit compter au moins valeur caractères.
@Positif Un champ numérique doit être strictement positif.
@Valide Un champ objet doit être validé récursivement : ses propres violations sont rapportées, préfixées du nom du champ porteur (par exemple adresse.rue). 
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.FIELD)
public @interface NonNul {
}

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.FIELD)
public @interface LongueurMin {
    int valeur();
}

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.FIELD)
public @interface Positif {
}

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.FIELD)
public @interface Valide {
}

Utilisation sur une classe de domaine :

class Personne {
    @NonNul
    @LongueurMin(valeur = 3)
    private String nom;

    @Positif
    private int age;

    @NonNul
    @Valide
    private Adresse adresse;
}

Le rapport de validation : Violation et RapportValidation

Violation associe le nom du champ fautif à un message ; RapportValidation agrège une liste de violations pour un objet entier.

class Violation {
    private final String champ;
    private final String message;
    // getChamp(), getMessage(), toString() -> "champ : message"
}

class RapportValidation {
    private final List<Violation> violations = new ArrayList<>();

    void ajouter(Violation violation) { violations.add(violation); }
    void ajouterToutes(List<Violation> nouvelles) { violations.addAll(nouvelles); }
    boolean estValide() { return violations.isEmpty(); }
    List<Violation> getViolations() { return Collections.unmodifiableList(violations); }
}

Un rapport sans violation (estValide() vrai) signifie que l'objet respecte toutes ses contraintes déclarées.

Le motif général d'une méthode de validation

Chaque contrainte suit le même schéma : trouver les champs annotés, lire leur valeur (avec setAccessible(true)), et ajouter une Violation si la règle est enfreinte.

List<Violation> validerNonNul(Object objet) {
    List<Violation> violations = new ArrayList<>();
    for (Field champ : trouverChampsAnnotes(objet.getClass(), NonNul.class)) {
        champ.setAccessible(true);
        Object valeur = champ.get(objet);
        if (valeur == null) {
            violations.add(new Violation(champ.getName(), "ne peut pas être nul"));
        }
    }
    return violations;
}

Une contrainte à élément, comme @LongueurMin, lit d'abord l'élément de l'annotation elle-même avant de le comparer à la valeur du champ :

int minimum = champ.getAnnotation(LongueurMin.class).valeur();

La validation récursive de @Valide réutilise la méthode de validation générale sur la valeur du champ, puis préfixe chaque nom de champ fautif du nom du champ porteur :

for (Field champ : trouverChampsAnnotes(objet.getClass(), Valide.class)) {
    champ.setAccessible(true);
    Object valeur = champ.get(objet);
    if (valeur != null) {
        for (Violation violation : valider(valeur).getViolations()) {
            violations.add(new Violation(
                champ.getName() + "." + violation.getChamp(),
                violation.getMessage()));
        }
    }
}

Un champ nul n'est pas de la responsabilité de @LongueurMin, @Positif ou @Valide : c'est @NonNul qui signale son absence, les autres contraintes se contentent de l'ignorer quand la valeur est nulle.

Exceptions courantes

Exception Cas typique Traitement habituel
ClassNotFoundException Class.forName(...) ne trouve pas la classe demandée. Vérifier le nom complet de la classe et le classpath.
NoSuchMethodException Aucune méthode ou aucun constructeur ne correspond au nom et aux types donnés. Vérifier le nom, la visibilité, et les types exacts des paramètres.
NoSuchFieldException Aucun champ ne correspond au nom donné. Vérifier le nom du champ et l'utilisation de getField ou getDeclaredField.
InstantiationException Impossible d'instancier la classe, par exemple parce qu'elle est abstraite. Créer une instance d'une classe concrète.
IllegalAccessException Le membre existe mais n'est pas accessible depuis le code courant. Utiliser un membre public ou appeler setAccessible(true) si c'est approprié.
IllegalArgumentException Arguments incompatibles lors d'un appel dynamique. Vérifier l'objet cible, le nombre d'arguments et leurs types.
InvocationTargetException La méthode ou le constructeur appelé a lui-même lancé une exception. Récupérer l'exception originale avec getCause().

InvocationTargetException

Throwable getCause()

Quand une méthode appelée par Method.invoke lance une exception, cette exception n'est pas lancée directement. Elle est enveloppée dans une InvocationTargetException.

try {
    methode.invoke(objet);
} catch (InvocationTargetException e) {
    Throwable cause = e.getCause();
    System.out.println(cause.getMessage());
}

Exemples neutres

Classe utilisée dans les exemples

class Produit {
    private String nom;
    private double prix;

    public Produit() {
        this("inconnu", 0.0);
    }

    public Produit(String nom, double prix) {
        this.nom = nom;
        this.prix = prix;
    }

    public String getNom() {
        return nom;
    }

    public void changerPrix(double nouveauPrix) {
        if (nouveauPrix < 0) {
            throw new IllegalArgumentException("prix négatif");
        }
        this.prix = nouveauPrix;
    }
}

Créer un objet puis appeler une méthode

Class<Produit> type = Produit.class;

Constructor<Produit> constructeur =
    type.getDeclaredConstructor(String.class, double.class);

Produit produit = constructeur.newInstance("Carnet", 4.5);

Method getNom = type.getDeclaredMethod("getNom");
String nom = (String) getNom.invoke(produit);

System.out.println(nom);

Filtrer des méthodes annotées

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
@interface Action {
    String nom();
}

class Controleur {
    @Action(nom = "ouvrir")
    public void ouvrir() {
        System.out.println("ouverture");
    }

    public void fermer() {
        System.out.println("fermeture");
    }
}

for (Method methode : Controleur.class.getDeclaredMethods()) {
    if (methode.isAnnotationPresent(Action.class)) {
        Action action = methode.getAnnotation(Action.class);
        System.out.println(action.nom() + " - " + methode.getName());
    }
}

Pièges fréquents

Problème Cause probable Correction
Une annotation n'est jamais détectée. La rétention n'est pas RUNTIME. Déclarer l'annotation avec @Retention(RetentionPolicy.RUNTIME).
getMethod ne trouve pas une méthode privée. getMethod ne cherche que les méthodes publiques accessibles. Utiliser getDeclaredMethod, puis éventuellement setAccessible(true).
NoSuchMethodException malgré un nom correct. Les types de paramètres ne correspondent pas exactement. Passer les classes exactes : int.class, String.class, double.class, etc.
IllegalArgumentException pendant invoke. Objet cible ou arguments incorrects. Vérifier que l'objet est bien une instance de la classe qui déclare la méthode, et que les arguments ont les bons types.
Une exception applicative est cachée dans InvocationTargetException. La méthode appelée dynamiquement a lancé une exception. Lire e.getCause() pour obtenir l'exception originale.
Le résultat de invoke doit être casté. invoke retourne toujours Object. Caster vers le type attendu après avoir vérifié la signature de la méthode.
Le programme dépend de l'ordre des méthodes retournées. La réflexion ne garantit pas l'ordre de déclaration. Trier explicitement si un ordre est nécessaire, par exemple avec Comparator.comparing(Method::getName).
