Classes, objets, encapsulation

Une classe décrit un type d'objet : ses attributs (état) et ses méthodes (comportement). Un objet est une instance de la classe, créée avec new.

public class Compte {

    private String numero;   // attribut d'instance, encapsulé
    private double solde;

    public String getNumero() {
        return numero;
    }

    public double getSolde() {
        return solde;
    }
}

Encapsulation : les attributs sont private ; l'accès extérieur passe par des getters (et éventuellement des setters). Cela permet de contrôler les modifications et de protéger les invariants de la classe.

Un setter n'est fourni que s'il est réellement nécessaire. Moins la classe expose de moyens de modifier son état, plus elle est facile à raisonner et à tester.

Constructeurs et surcharge

Le constructeur initialise l'objet à sa création. Il porte le nom de la classe et n'a pas de type de retour.

public NomClasse(paramètres) { ... } 
public Compte(String numero, double soldeInitial) {
    if (numero == null || numero.isBlank()) {
        throw new IllegalArgumentException("numero obligatoire");
    }
    this.numero = numero;
    this.solde = soldeInitial;
}

Surcharge et chaînage avec this(...)

Une classe peut définir plusieurs constructeurs (surcharge). Un constructeur peut en appeler un autre avec this(...), qui doit être la première instruction.

public Compte(String numero) {
    this(numero, 0);   // délègue au constructeur complet
}
Valider les paramètres dans le constructeur le plus complet et y faire déléguer les autres : la validation n'est écrite qu'une seule fois.

Attributs et méthodes statiques

Un membre static appartient à la classe, pas aux instances : une seule copie partagée.

public class Commande {

    private static int compteur = 0;   // partagé par toutes les commandes

    private final int numero;

    public Commande() {
        compteur++;
        this.numero = compteur;   // numérotation automatique
    }

    public static int getNombreDeCommandes() {
        return compteur;
    }
}

Usage typique : numérotation automatique des instances, constantes (static final), méthodes utilitaires.

Constantes dans une interface

Un champ déclaré dans une interface est implicitement public static final, même sans écrire ces mots-clés. On accède donc directement à la constante via le nom de l'interface, sans instance.

public interface Couleurs {
    public static final String ROUGE = "rouge";   // public static final implicite
    public static final String BLEU = "bleu";
}
Couleurs.ROUGE   // accès direct, pas besoin d'instance

Héritage et classes abstraites

Une sous-classe extends sa superclasse : elle hérite des attributs et méthodes non privés, peut en ajouter et peut redéfinir (@Override) les méthodes héritées.

public abstract class Vehicule {

    private final String immatriculation;

    protected Vehicule(String immatriculation) {
        this.immatriculation = immatriculation;
    }

    public String getImmatriculation() {
        return immatriculation;
    }

    // Méthode abstraite : pas de corps, chaque sous-classe DOIT l'implémenter.
    public abstract double calculerTaxe();
}

public class Voiture extends Vehicule {

    private final int puissance;

    public Voiture(String immatriculation, int puissance) {
        super(immatriculation);   // appel au constructeur parent, première instruction
        this.puissance = puissance;
    }

    @Override
    public double calculerTaxe() {
        return puissance * 12.5;
    }
}
  • Une classe abstract ne peut pas être instanciée directement ; elle sert de base commune.
  • super(...) appelle le constructeur parent et doit être la première instruction du constructeur.
  • @Override est fortement recommandé : le compilateur vérifie qu'on redéfinit bien une méthode existante.
  • Le polymorphisme permet de manipuler une Voiture via une référence de type Vehicule : l'appel v.calculerTaxe() exécute la version de la classe réelle de l'objet.

Égalité : ==, equals, hashCode

== compare les références (même objet en mémoire). equals compare le contenu, si la classe le redéfinit — sinon il hérite du comportement de Object, qui équivaut à ==.

@Override
public boolean equals(Object o) {
    if (this == o) {
        return true;
    }
    if (o == null || getClass() != o.getClass()) {
        return false;
    }
    Compte compte = (Compte) o;
    return numero.equals(compte.numero);
}

@Override
public int hashCode() {
    return numero.hashCode();
}
Contrat : deux objets égaux selon equals doivent avoir le même hashCode. Toujours redéfinir les deux ensemble, sinon les collections basées sur le hachage (HashSet, HashMap) se comportent de façon incohérente.

IntelliJ génère les deux méthodes : Alt+Insert → equals() and hashCode().

Association entre objets

Un objet peut référencer un autre objet ou une collection d'objets, selon la multiplicité UML de l'association : 1 (exactement un, attribut simple), 0..1 (zéro ou un, attribut pouvant valoir null) ou * (zéro ou plusieurs, une collection comme ArrayList). Dans une association bidirectionnelle, chacun référence l'autre : la cohérence des deux côtés se gère dans les méthodes, pas depuis l'extérieur.

public class Banque {

    private List<Compte> comptes = new ArrayList<>();

    public boolean ajouterCompte(Compte compte) {
        if (compte == null || comptes.contains(compte)) {
            return false;
        }
        return comptes.add(compte);
    }

    // Copie défensive : l'appelant ne peut pas modifier la liste interne.
    public List<Compte> getComptes() {
        return new ArrayList<>(comptes);
    }
}
Pattern courant pour une méthode booléenne qui peut échouer (ajouterCompte ci-dessus, inscrire plus bas) : renvoyer false immédiatement si une condition bloque l'opération, sans rien modifier ; sinon effectuer l'opération et renvoyer true. L'appelant peut alors tester le résultat (if (!banque.ajouterCompte(c)) { ... }).

Interface Iterable et méthode iterator

Pour permettre de parcourir une collection interne sans exposer l'objet List lui-même (donc sans permettre d'y ajouter/retirer des éléments depuis l'extérieur), la classe implémente Iterable<T> et fournit une méthode iterator qui renvoie l'itérateur de sa liste interne.

public class Bibliotheque implements Iterable<Livre> {

    private ArrayList<Livre> livres = new ArrayList<>();

    @Override
    public Iterator<Livre> iterator() {
        return livres.iterator();
    }
}

Import nécessaire : java.util.Iterator. L'encapsulation est préservée : le code extérieur peut lire les livres via un for, mais ne peut pas modifier la liste interne (contrairement à un getter qui renverrait directement livres).

Parcours avec foreach

Le foreach fonctionne sur tout objet Iterable (donc aussi sur toute ArrayList, et sur Bibliotheque ci-dessus) : Java appelle sa méthode iterator en coulisses.

for (Compte compte : comptes) {
    System.out.println(compte.getNumero());
}

for (Livre livre : bibliotheque) {   // bibliotheque implémente Iterable<Livre>
    System.out.println(livre);
}

IllegalArgumentException dans un constructeur

Quand un paramètre est invalide, le constructeur lance une exception plutôt que de créer un objet incohérent :

if (montant <= 0) {
    throw new IllegalArgumentException("montant strictement positif attendu");
}

IllegalArgumentException est une exception non vérifiée (unchecked) : pas de throws à déclarer.

UnsupportedOperationException

Deuxième exemple canonique d'exception unchecked (hérite elle aussi de RuntimeException) : elle signale qu'une opération existe mais n'est pas autorisée dans ce contexte précis — par exemple une méthode volontairement non implémentée, ou une modification refusée sur une liste protégée.

throw new UnsupportedOperationException("Cette opération n'est pas autorisée");

Construire une chaîne — toString

@Override
public String toString() {
    return "Compte " + numero + " (solde : " + solde + ")";
}

Pour des concaténations répétées dans une boucle, préférer StringBuilder (voir le cheat sheet semaine 2).

Exemple complet

import java.util.ArrayList;
import java.util.List;

public abstract class Personne {

    private static int compteur = 0;

    private final int id;
    private final String nom;

    protected Personne(String nom) {
        if (nom == null || nom.isBlank()) {
            throw new IllegalArgumentException("nom obligatoire");
        }
        compteur++;
        this.id = compteur;
        this.nom = nom;
    }

    public int getId() { return id; }
    public String getNom() { return nom; }

    public abstract String getRole();

    @Override
    public String toString() {
        return getRole() + " " + nom + " (id " + id + ")";
    }
}

public class Etudiant extends Personne {

    private final List<String> cours = new ArrayList<>();

    public Etudiant(String nom) {
        super(nom);
    }

    public boolean inscrire(String intitule) {
        if (intitule == null || cours.contains(intitule)) {
            return false;
        }
        return cours.add(intitule);
    }

    public List<String> getCours() {
        return new ArrayList<>(cours);
    }

    @Override
    public String getRole() {
        return "Étudiant";
    }
}

Pièges fréquents

Problème Cause probable Correction
contains/remove ne trouve pas un objet « identique » equals non redéfini : comparaison par référence. Redéfinir equals et hashCode ensemble.
Comparaison de chaînes avec == == compare les références, pas le contenu. Utiliser equals : "a".equals(variable).
La liste interne est modifiée depuis l'extérieur Le getter renvoie la référence de la liste interne. Renvoyer une copie (new ArrayList<>(liste)) ou une vue non modifiable.
super(...) ne compile pas Pas en première instruction du constructeur. Placer super(...) (ou this(...)) en tout premier.
Objet créé avec un état invalide Pas de validation des paramètres du constructeur. Lancer IllegalArgumentException dès qu'un paramètre est invalide.
Un attribut statique garde sa valeur entre les tests static = partagé par toute la JVM, pas réinitialisé par instance. Éviter l'état statique mutable ; le réinitialiser explicitement si nécessaire.
UnsupportedOperationException à l'exécution Opération appelée alors qu'elle n'est pas autorisée dans ce contexte (ex. méthode volontairement non implémentée). Vérifier que l'opération est bien censée être disponible ; sinon la retirer de l'appel ou l'implémenter.
