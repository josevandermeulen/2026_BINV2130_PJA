Thread et Runnable

Un thread est un fil d'exécution indépendant. Le code à exécuter s'exprime via l'interface fonctionnelle Runnable.

Thread(Runnable target) void start() void run() void join() throws InterruptedException static void sleep(long millis) throws InterruptedException static Thread currentThread() 
Thread t = new Thread(() -> {
    System.out.println("dans " + Thread.currentThread().getName());
});
t.start();     // démarre le thread : run() s'exécute en parallèle
t.join();      // attend la fin du thread
start(), jamais run(). Appeler run() directement exécute le code dans le thread courant, séquentiellement — aucun parallélisme. Seul start() crée un nouveau fil d'exécution.

Race conditions et synchronized

Quand plusieurs threads modifient la même donnée, les opérations s'entrelacent de façon imprévisible. Même compteur++ n'est pas atomique (lecture, incrément, écriture) : deux threads peuvent lire la même valeur et l'un des deux incréments est perdu. C'est une race condition : le résultat dépend de l'ordre d'exécution.

public class Compteur {

    private int valeur = 0;

    // synchronized : un seul thread à la fois peut exécuter cette méthode
    public synchronized void incrementer() {
        valeur++;
    }

    public synchronized int getValeur() {
        return valeur;
    }
}

synchronized pose un verrou sur l'objet : les autres threads attendent leur tour. On peut aussi synchroniser un bloc précis :

synchronized (verrou) {
    // section critique
}

Deadlock (interblocage)

Un deadlock survient quand deux threads (ou plus) s'attendent mutuellement, chacun détenant un verrou dont l'autre a besoin. Plus aucun des deux ne peut avancer : le programme est figé, sans exception, sans message d'erreur.

Le scénario classique demande deux verrous pris dans un ordre différent par deux threads :

Object verrouA = new Object();
Object verrouB = new Object();

// Thread 1
synchronized (verrouA) {          // 1. prend A
    Thread.sleep(50);
    synchronized (verrouB) {      // 3. attend B... détenu par le thread 2
        // ...
    }
}

// Thread 2
synchronized (verrouB) {          // 2. prend B
    Thread.sleep(50);
    synchronized (verrouA) {      // 4. attend A... détenu par le thread 1
        // ...
    }
}

Le thread 1 détient verrouA et attend verrouB ; le thread 2 détient verrouB et attend verrouA. Aucun des deux ne relâchera jamais ce que l'autre attend.

Quatre conditions doivent être réunies simultanément (conditions de Coffman) pour qu'un deadlock soit possible :

  • Exclusion mutuelle : les ressources (verrous) ne peuvent être détenues que par un thread à la fois ;
  • Rétention et attente : un thread garde ses verrous pendant qu'il en attend d'autres ;
  • Pas de préemption : un verrou ne peut pas être retiré de force à son détenteur ;
  • Attente circulaire : chaque thread attend une ressource détenue par le suivant, en boucle fermée.

Casser une seule de ces conditions suffit à rendre le deadlock impossible. En pratique, la parade la plus simple casse l'attente circulaire : toujours acquérir les verrous dans le même ordre global. Autres bonnes habitudes :

  • garder les sections synchronisées courtes et éviter d'y appeler du code qui prend lui-même des verrous ;
  • ne jamais faire d'appel bloquant (sleep, entrée/sortie, join...) en détenant un verrou ;
  • préférer un seul verrou à plusieurs quand c'est possible — pas de deuxième verrou, pas de cycle.
Contrairement à une race condition (résultat faux, mais le programme continue), un deadlock se voit : le programme reste suspendu, sans erreur. Un thread dump (jstack, ou le bouton de dump de threads d'IntelliJ) montre alors quels threads se bloquent mutuellement et sur quels verrous.

ExecutorService

Créer un thread par tâche ne passe pas à l'échelle. Un ExecutorService gère un pool de threads réutilisés.

Fabriques : java.util.concurrent.Executors.

static ExecutorService newFixedThreadPool(int nThreads) static ExecutorService newCachedThreadPool() static ExecutorService newSingleThreadExecutor()

Méthodes principales :

void execute(Runnable command) <T> Future<T> submit(Callable<T> task) void shutdown() 
ExecutorService executor = Executors.newFixedThreadPool(4);

executor.execute(() -> traiterCommande("CMD-1"));

executor.shutdown();   // n'accepte plus de tâches, termine celles en cours
Sans shutdown(), les threads du pool restent vivants et le programme ne se termine pas.

CompletableFuture - création

Classe : java.util.concurrent.CompletableFuture<T>. Représente un résultat futur, produit de façon asynchrone, sur lequel on peut chaîner des traitements.

static CompletableFuture<Void> runAsync(Runnable runnable) static <U> CompletableFuture<U> supplyAsync(Supplier<U> supplier) static <U> CompletableFuture<U> supplyAsync(Supplier<U> supplier, Executor executor) static <U> CompletableFuture<U> completedFuture(U value) 
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
    // exécuté dans un thread du pool commun (ForkJoinPool.commonPool)
    return apiClient.chargerProfil("alice");
});

Sans executor explicite, les tâches tournent dans le pool commun de la JVM. Passer son propre Executor donne le contrôle du nombre de threads.

CompletableFuture - composition

Transformer le résultat : thenApply

<U> CompletableFuture<U> thenApply(Function<T, U> fn)

Applique une fonction synchrone au résultat : T → U.

CompletableFuture<Integer> longueur =
    future.thenApply(profil -> profil.length());

Consommer le résultat : thenAccept, thenRun

CompletableFuture<Void> thenAccept(Consumer<T> action) CompletableFuture<Void> thenRun(Runnable action) 
future.thenAccept(profil -> System.out.println(profil));

Chaîner une autre opération asynchrone : thenCompose

<U> CompletableFuture<U> thenCompose(Function<T, CompletableFuture<U>> fn)

Quand l'étape suivante renvoie elle-même un CompletableFuture (autre appel asynchrone). thenCompose aplatit le résultat :

CompletableFuture<List<Commande>> commandes =
    chargerUtilisateur("alice")                       // CF<Utilisateur>
        .thenCompose(u -> chargerCommandes(u.getId())); // CF<List<Commande>>
thenApply avec une fonction qui renvoie un CompletableFuture produit un CompletableFuture<CompletableFuture<U>> imbriqué. Pour chaîner de l'asynchrone, utiliser thenCompose.

Combiner deux futures indépendants : thenCombine

<U,V> CompletableFuture<V> thenCombine(CompletionStage<U> other, BiFunction<T, U, V> fn)

Les deux opérations tournent en parallèle ; la fonction fusionne leurs deux résultats quand les deux sont prêts :

CompletableFuture<Facture> facture =
    chargerClient("alice")
        .thenCombine(chargerTarifs(), (client, tarifs) -> calculer(client, tarifs));

Gérer les erreurs : exceptionally, handle

CompletableFuture<T> exceptionally(Function<Throwable, T> fn) <U> CompletableFuture<U> handle(BiFunction<T, Throwable, U> fn) 
future.exceptionally(ex -> "profil par défaut");

Attendre plusieurs futures

Récupérer les résultats

T get() throws InterruptedException, ExecutionException T get(long timeout, TimeUnit unit) T join() boolean isDone()

get() et join() bloquent le thread courant jusqu'au résultat. join() lance une exception non vérifiée (CompletionException), get() des exceptions vérifiées.

CompletableFuture<String> f = CompletableFuture.supplyAsync(() -> calculLong());
// ... autre travail en parallèle ...
String resultat = f.join();   // bloque ici si pas encore fini
Bloquer avec get/join annule l'intérêt de l'asynchrone si on le fait immédiatement. Préférer chaîner (thenApply, thenAccept…) et ne bloquer qu'en bout de chaîne (par exemple dans le main pour attendre la fin).

Exemple complet neutre

import java.util.concurrent.CompletableFuture;

public class Demo {

    public static void main(String[] args) {
        CompletableFuture<String> commande =
            chargerUtilisateur("alice")                          // async
                .thenCompose(Demo::chargerDerniereCommande)      // async chaîné
                .thenCombine(chargerTauxTva(),                   // async parallèle
                    (cmd, tva) -> cmd + " (TVA " + tva + "%)")
                .thenApply(String::toUpperCase)                  // transformation
                .exceptionally(ex -> "ERREUR : " + ex.getMessage());

        System.out.println("Le main continue pendant le chargement...");

        System.out.println(commande.join());   // on ne bloque qu'à la fin
    }

    static CompletableFuture<String> chargerUtilisateur(String nom) {
        return CompletableFuture.supplyAsync(() -> {
            attendre(300);                     // simule un appel réseau
            return nom;
        });
    }

    static CompletableFuture<String> chargerDerniereCommande(String utilisateur) {
        return CompletableFuture.supplyAsync(() -> "commande de " + utilisateur);
    }

    static CompletableFuture<Integer> chargerTauxTva() {
        return CompletableFuture.supplyAsync(() -> 21);
    }

    static void attendre(long millis) {
        try {
            Thread.sleep(millis);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}

Pièges fréquents

Problème Cause probable Correction
Aucun parallélisme constaté run() appelé au lieu de start(). Appeler start() sur le thread.
Compteur final incorrect (aléatoirement) Race condition : opération non atomique partagée. synchronized sur la section critique (ou AtomicInteger).
Le programme ne se termine jamais ExecutorService jamais arrêté. Appeler shutdown() après soumission des tâches.
Le main se termine avant les traitements async Les threads du pool commun sont daemon : la JVM n'attend pas. join() sur le future final avant de quitter le main.
CompletableFuture<CompletableFuture<T>> thenApply utilisé avec une fonction asynchrone. Utiliser thenCompose.
Résultat disponible mais jamais consommé Chaîne construite mais future final ignoré. Chaîner un thenAccept, ou join() en bout de chaîne.

// Thread
Thread t = new Thread(() -> travail());
t.start();  t.join();
Thread.sleep(1000);

// Synchronisation
public synchronized void incrementer() { valeur++; }

// ExecutorService
ExecutorService ex = Executors.newFixedThreadPool(4);
ex.execute(() -> tache());
ex.shutdown();

// CompletableFuture
CompletableFuture.supplyAsync(() -> produireValeur())   // CF<T>
    .thenApply(v -> transformer(v))                     // T → U (sync)
    .thenCompose(v -> autreAsync(v))                    // T → CF<U> (async)
    .thenCombine(autreCF, (a, b) -> fusionner(a, b))    // 2 CF parallèles
    .thenAccept(v -> consommer(v))                      // T → void
    .exceptionally(ex2 -> valeurParDefaut);             // gestion d'erreur
