Principe du TDD

Le Test Driven Development inverse l'ordre habituel : on écrit le test avant le code. Le test décrit le comportement attendu ; le code n'existe que pour le faire passer.

  • Le test sert de spécification exécutable : il documente ce que la méthode doit faire.
  • Le code produit est couvert par construction : chaque comportement a son test.
  • La conception émerge des besoins réels (l'API est conçue du point de vue de l'appelant).
TDD ne veut pas dire « écrire tous les tests puis tout le code » : on avance un scénario à la fois, en boucle courte.

Spécifier les scénarios de tests

Avant de coder, on identifie les cas d'utilisation (UC) de l'application, puis pour chaque UC les scénarios de tests :

  • Cas positifs : l'opération réussit — vérifier le résultat et le nouvel état.
  • Cas négatifs : l'opération échoue proprement (retour false, exception) — vérifier aussi que l'état n'a pas changé.
  • Cas limites : valeurs frontières (0, valeur minimale, chaîne vide, null, doublon, élément inexistant…).
  • Classes d'équivalence : pour un même paramètre, regrouper les valeurs possibles en catégories qui doivent produire un comportement similaire ; ne tester qu'un représentant par catégorie plutôt que de multiplier les cas redondants.

Exemple de spécification pour « Ajouter une tâche à la liste » :

1. (UC1) Ajouter une tâche à la liste :
   1.1 addTask         : la tâche est contenue dans la liste,
                         on informe du succès de l'opération
   1.2 addEmptyTask    : tâche vide ou nulle, non ajoutée,
                         on informe de l'échec
   1.3 addExistingTask : tâche déjà présente, on informe de l'échec

Chaque scénario devient une méthode de test au nom parlant (addTask, addEmptyTask…). La liste des scénarios est le plan de travail du TDD.

Classes d'équivalence : un exemple

Pour le paramètre task de addTask, on peut regrouper les valeurs possibles en deux classes, chacune censée produire le même comportement :

Classe « nom vide »   : "", "   ", null   → comportement attendu : échec (un seul scénario addEmptyTask)
Classe « nom valide » : "tâche 1", "abc"  → comportement attendu : succès (un seul scénario addTask)

Plutôt que d'écrire un scénario distinct pour la chaîne vide, la chaîne de blancs et null, un seul scénario addEmptyTask couvre un représentant de chaque valeur de la classe « nom vide » dans le même assertAll (voir l'exemple de la section suivante) — inutile de multiplier les scénarios pour des entrées qui, du point de vue du code, doivent toutes échouer de la même façon.

Le cycle red-green-refactor

Pour chaque scénario, trois étapes strictes :

1. Red — écrire un test qui échoue

Travailler à l'envers, à partir des assertions : écrire d'abord ce qu'on veut vérifier, pour une méthode et parfois une classe qui n'existent pas encore. Laisser l'IDE générer les éléments manquants.

@Test
void removeTask() {
    todoList.addTask("tâche 1");
    assertAll(
        () -> assertTrue(todoList.removeTask("tâche 1")),   // n'existe pas encore
        () -> assertFalse(todoList.containsTask("tâche 1"))
    );
}
Le test doit échouer pour de bonnes raisons. Exécuter le test et vérifier qu'il échoue sur l'assertion attendue (pas sur une erreur de compilation résiduelle ou une exception imprévue). Un test jamais vu en échec ne prouve rien : il pourrait passer pour de mauvaises raisons.

2. Green — écrire le code minimum

Écrire une implémentation évidente qui fait passer le test — sans anticiper les scénarios suivants, mais sans écrire du code volontairement faux (pas de valeur codée en dur juste pour tricher avec le test).

public boolean removeTask(String task) {
    return tasks.remove(task);
}

3. Refactor — améliorer sans changer le comportement

Nettoyer le code (noms, duplications, structure) en relançant les tests après chaque changement. Règle de trois : dès qu'une duplication apparaît trois fois, la remplacer par du code réutilisable ; avant cela, laisser tel quel.

Tous les tests verts après le refactor = le comportement est préservé. C'est le filet de sécurité du TDD.

Outillage IntelliJ pour le TDD

IntelliJ génère le code manquant à partir du test — c'est ce qui rend le « travail à l'envers » fluide :

Besoin Action IntelliJ
Créer une méthode de test Curseur dans la classe de test : Alt+Insert → Test Method
Créer la classe manquante référencée par le test Curseur sur le nom en rouge : Alt+Entrée → Create class
Créer la méthode manquante Curseur sur l'appel en rouge : Alt+Entrée → Create method
Ajouter la bibliothèque JUnit5 Cliquer une fois sur Fix quand « JUnit5 library not found » s'affiche
Lancer tous les tests Clic droit sur le dossier tests → Run 'All Tests'

TDD sur du code existant

Quand une exigence change (par exemple : une tâche n'est plus une String mais un objet avec titre et description), le cycle s'applique à partir des tests existants :

  1. Adapter les tests d'abord — mettre à jour les scénarios existants pour la nouvelle exigence. Le code ne compile plus ou échoue : c'est le red, pour de bonnes raisons.
  2. Mettre à jour le code minimum — faire passer tous les tests, anciens adaptés comme nouveaux. Le green garantit que rien d'existant n'est cassé.
  3. Refactorer sous la protection des tests.

Créer de nouveaux tests quand la modification introduit : de nouveaux comportements (nouvelles UC), de nouveaux cas limites (ex. l'égalité de deux objets distincts de même contenu — un « clone »), de nouvelles règles de validation (constructeur qui rejette un titre vide).

Si un ancien scénario n'a plus de sens avec la nouvelle exigence, c'est la spécification qui a changé : mettre à jour le scénario, ne pas le supprimer silencieusement.

TDD assisté par IA

Les assistants de code basés sur l'IA (Claude Code, GitHub Copilot…) peuvent générer en une seule réponse un test et son implémentation. Cela va à l'encontre du principe même du TDD : si le test et le code arrivent ensemble, on ne voit jamais le test échouer, et on perd le bénéfice principal du cycle — la confirmation que le test vérifie bien ce qu'on croit vérifier.

Pour tirer parti de l'IA sans perdre les bénéfices du TDD, on continue à respecter le cycle red-green-refactor étape par étape, même quand une IA écrit le code à la place du développeur :

  1. Red : demander à l'IA de générer uniquement le scénario de test (pas l'implémentation). L'exécuter et vérifier qu'il échoue pour de bonnes raisons — les mêmes vérifications que pour un test écrit à la main s'appliquent.
  2. Green : demander ensuite à l'IA d'écrire le code minimal pour faire passer ce test précis (pas les scénarios suivants). Exécuter tous les tests et vérifier qu'ils passent.
  3. Refactor : demander à l'IA d'appliquer la règle de trois si de la duplication est apparue. Relancer les tests pour confirmer qu'ils passent toujours.
Ne jamais valider un test et son implémentation générés ensemble sans être passé par l'étape red. Un test qui n'a jamais échoué peut ne rien tester (mauvais setup, assertion toujours vraie…) — le risque est le même qu'avec du code écrit à la main, mais plus facile à manquer quand tout est généré d'un coup : l'IA peut écrire un test qui colle exactement à l'implémentation qu'elle va produire, masquant ainsi les vrais bugs de compréhension du besoin.

Sur du code existant

Le même risque existe lors d'une évolution, avec une variante : quand on demande à une IA d'adapter un test existant à une nouvelle exigence, elle peut être tentée de mettre à jour le test et le code de production en même temps. On perd alors la vérification essentielle du cycle — voir le test adapté échouer pour de bonnes raisons avant de toucher au code.

Demander d'abord uniquement la mise à jour du test, l'exécuter et constater l'échec, puis seulement ensuite demander la mise à jour du code.

Exemple complet neutre

Itération TDD complète sur un scénario « retirer un article absent du panier » :

// ── RED : le test d'abord, il échoue ─────────────────────────
@Test
void retirerArticleAbsentEchoue() {
    Panier panier = new Panier();
    panier.ajouter("livre");

    assertAll(
        () -> assertFalse(panier.retirer("stylo")),   // retirer() n'existe pas encore
        () -> assertTrue(panier.contient("livre"))    // l'état n'a pas changé
    );
}

// ── GREEN : le code minimum ──────────────────────────────────
public boolean retirer(String article) {
    return articles.remove(article);
}

// ── REFACTOR : rien à nettoyer ici, on passe au scénario suivant.

Puis on recommence le cycle avec le scénario suivant de la spécification (retrait réussi, retrait sur panier vide…).

Pièges fréquents

Problème Cause probable Correction
Le test passe du premier coup Le comportement existait déjà, ou le test ne teste rien. Vérifier l'assertion (la fausser volontairement doit faire échouer le test), ou passer au scénario suivant.
Test jamais vu en échec Étape red sautée. Toujours exécuter le test avant d'implémenter et constater l'échec attendu.
Code qui anticipe des besoins futurs Implémentation au-delà du « minimum » du green. N'écrire que ce que le test courant exige ; les scénarios suivants amèneront le reste.
Cas négatif qui ne vérifie que le retour Oubli de vérifier l'état après l'échec. Vérifier aussi que l'état n'a pas changé (l'élément est toujours/toujours pas présent).
Refactor qui casse un test Changement de comportement déguisé en nettoyage. Relancer les tests après chaque étape de refactor ; revenir en arrière si un test tombe.
Gros scénarios fourre-tout Plusieurs comportements testés dans une méthode. Un scénario = une méthode de test au nom parlant.

Référence rapide

Démarche TDD :

1. Identifier les UC et leurs scénarios de tests
   (cas positifs, négatifs, limites).

2. Pour chaque scénario :
   RED      écrire le test (à partir des assertions),
            l'exécuter, le voir échouer pour de bonnes raisons.
   GREEN    écrire le code minimum, évident, qui le fait passer.
   REFACTOR nettoyer (règle de trois pour les duplications),
            tous les tests restent verts.

3. Code existant qui évolue :
   adapter les tests d'abord (red) → adapter le code (green)
   → refactorer ; ajouter des tests pour tout nouveau
   comportement, cas limite ou règle de validation.
