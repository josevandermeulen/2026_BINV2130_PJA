Énumérés

Un enum définit un type dont les valeurs possibles sont fixées à la compilation. Chaque constante est une instance unique.

public enum TypePromo {
    SOLDE, PUB, DESTOCKAGE
}

Méthodes disponibles sur tout enum

String name() int ordinal() static T[] values() static T valueOf(String name) 
TypePromo promo = TypePromo.SOLDE;
promo.name();                    // "SOLDE"
TypePromo.values();              // [SOLDE, PUB, DESTOCKAGE]
TypePromo.valueOf("PUB");        // TypePromo.PUB (exception si inconnu)

Attributs et méthodes dans un enum

public enum Jour {
    LUNDI("Lun"), MARDI("Mar");

    private final String abreviation;

    Jour(String abreviation) {          // constructeur toujours privé
        this.abreviation = abreviation;
    }

    public String getAbreviation() {
        return abreviation;
    }
}

Switch sur un enum

Style classique :

switch (promo) {
    case SOLDE:
        System.out.println("Soldes !");
        break;
    case PUB:
    case DESTOCKAGE:
        System.out.println("Autre promo");
        break;
}

Style flèche (Java 14+), plus concis, plusieurs constantes sur un même case :

switch (promo) {
    case SOLDE -> System.out.println("Soldes !");
    case PUB, DESTOCKAGE -> System.out.println("Autre promo");
}
Comparer deux énumérés avec == est correct et idiomatique : chaque constante n'existe qu'en un seul exemplaire.

List et Set

Package java.util. Deux interfaces principales de Collection :

  • List<E> : séquence ordonnée, doublons autorisés, accès par index. Implémentation courante : ArrayList.
  • Set<E> : pas de doublons (au sens de equals), pas d'accès par index. Implémentations : HashSet (rapide, ordre non garanti), TreeSet (trié).

Méthodes principales (communes)

boolean add(E e) boolean remove(Object o) boolean contains(Object o) int size() boolean isEmpty() void clear()

Spécifique à List

E get(int index) E set(int index, E element) void add(int index, E element) E remove(int index) int indexOf(Object o) 
List<String> noms = new ArrayList<>();
noms.add("alice");
noms.add("alice");        // ok, doublon accepté

Set<String> unique = new HashSet<>();
unique.add("alice");
unique.add("alice");      // renvoie false, pas de doublon
Dans les collections HashXXX (HashSet, HashMap), il faut redéfinir hashCode et equals dans la classe des éléments (ou des clés). Ces implémentations s'appuient sur les deux méthodes : sans redéfinition, les doublons ne sont pas détectés et les recherches échouent. Toujours redéfinir les deux ensemble, de manière cohérente (deux objets equals doivent avoir le même hashCode).

Map et parcours

Map<K, V> associe des clés uniques à des valeurs. Implémentations : HashMap (ordre non garanti), TreeMap (clés triées). Pour une HashMap, la classe des clés doit redéfinir hashCode et equals.

V put(K key, V value) V get(Object key) V getOrDefault(Object key, V defaultValue) boolean containsKey(Object key) V remove(Object key) Set<K> keySet() Collection<V> values() Set<Map.Entry<K, V>> entrySet() 
Map<String, Integer> stock = new HashMap<>();
stock.put("livre", 3);
stock.put("livre", 5);          // remplace la valeur existante
stock.get("stylo");             // null si absent
stock.getOrDefault("stylo", 0); // 0 si absent

Parcourir une Map

// Par entrées (clé + valeur en même temps)
for (Map.Entry<String, Integer> entry : stock.entrySet()) {
    System.out.println(entry.getKey() + " : " + entry.getValue());
}

// Par clés
for (String cle : stock.keySet()) { ... }

// Par valeurs
for (Integer valeur : stock.values()) { ... }
put sur une clé existante remplace la valeur et renvoie l'ancienne. C'est le comportement attendu, par exemple, quand on redéfinit un prix pour une quantité déjà présente.

Collections triées : SortedSet, SortedMap

TreeSet implémente SortedSet ; TreeMap implémente SortedMap. Les éléments (ou les clés) sont maintenus triés en permanence.

E first() / K firstKey() E last() / K lastKey() SortedMap<K,V> headMap(K toKey) SortedMap<K,V> tailMap(K fromKey)

Le tri utilise soit l'ordre naturel des éléments (interface Comparable), soit un Comparator passé au constructeur :

// Ordre naturel (String implémente Comparable)
SortedSet<String> noms = new TreeSet<>();

// Ordre décroissant via un Comparator
SortedMap<LocalDate, Prix> historique =
    new TreeMap<>(Comparator.reverseOrder());
Dans un TreeSet/TreeMap, c'est la comparaison (pas equals) qui détermine les doublons : deux éléments dont la comparaison vaut 0 sont considérés identiques. Garder la comparaison cohérente avec equals.

Comparable et Comparator

Ordre naturel : Comparable<T>

La classe elle-même définit comment ses instances s'ordonnent.

int compareTo(T o)

Contrat : négatif si this avant o, 0 si équivalents, positif si après.

public class Produit implements Comparable<Produit> {

    @Override
    public int compareTo(Produit autre) {
        int parNom = this.nom.compareTo(autre.nom);
        if (parNom != 0) {
            return parNom;
        }
        int parMarque = this.marque.compareTo(autre.marque);
        if (parMarque != 0) {
            return parMarque;
        }
        return this.rayon.compareTo(autre.rayon);
    }
}

Ordre externe : Comparator<T>

Un objet à part qui compare deux instances — utile pour des ordres multiples ou quand on ne peut pas modifier la classe.

int compare(T o1, T o2) static <T extends Comparable> Comparator<T> naturalOrder() static <T extends Comparable> Comparator<T> reverseOrder() static <T, U extends Comparable> Comparator<T> comparing(Function<T, U> keyExtractor) default Comparator<T> thenComparing(Comparator<T> other) default Comparator<T> reversed() 
Comparator<Produit> parMarque = Comparator.comparing(Produit::getMarque)
                                          .thenComparing(Produit::getNom);

List<Produit> produits = ...;
produits.sort(parMarque);
Collections.sort(produits);      // ordre naturel (Comparable)

Classes anonymes

Une classe anonyme implémente une interface (ou étend une classe) sur place, sans être nommée. Syntaxe historique fréquente pour les Comparator :

Comparator<Produit> parNom = new Comparator<Produit>() {
    @Override
    public int compare(Produit p1, Produit p2) {
        return p1.getNom().compareTo(p2.getNom());
    }
};

Depuis Java 8, une interface fonctionnelle (une seule méthode abstraite) peut s'écrire en lambda, plus concise :

Comparator<Produit> parNom = (p1, p2) -> p1.getNom().compareTo(p2.getNom());

Renvoyer une collection non modifiable

Pour exposer une collection interne sans permettre sa modification :

static <T> List<T> List.copyOf(Collection<T> coll) static <T> Set<T> Set.copyOf(Collection<T> coll) static <T> List<T> Collections.unmodifiableList(List<T> list) static <K,V> SortedMap<K,V> Collections.unmodifiableSortedMap(SortedMap<K,V> m) 
public List<Produit> getProduits() {
    return Collections.unmodifiableList(produits);
}

Différence : List.copyOf fait une copie figée (indépendante des changements ultérieurs) ; unmodifiableList est une vue : elle reflète les changements internes mais interdit la modification via cette référence (lance UnsupportedOperationException).

La théorie de la semaine ne couvre que Collections.unmodifiableXXX. List.copyOf/Set.copyOf sont ajoutés ici en complément (pas faux, mais au-delà du programme officiel de cette semaine).

StringBuilder, autoboxing, Duration

StringBuilder

Concaténer des String dans une boucle crée un objet par itération. StringBuilder accumule efficacement :

StringBuilder append(...) String toString() 
StringBuilder sb = new StringBuilder();
for (Produit p : produits) {
    sb.append(p.getNom()).append("\n");
}
return sb.toString();

Wrapper classes et autoboxing

Les collections ne stockent que des objets : Integer, Double, Boolean… La conversion primitif ↔ wrapper est automatique (autoboxing/unboxing).

List<Integer> nombres = new ArrayList<>();
nombres.add(3);            // int → Integer automatique
int premier = nombres.get(0);   // Integer → int automatique
Unboxing d'un null → NullPointerException. Exemple : int n = map.get("absent"); plante si la clé n'existe pas.

Duration

static Duration ofMinutes(long minutes) static Duration between(Temporal start, Temporal end) Duration plusMinutes(long minutes) Duration minusMinutes(long minutes) long toHours() long toMinutes() int toMinutesPart() 
Duration duree = Duration.ofMinutes(90);
duree.toHours();         // 1  (heures complètes)
duree.toMinutesPart();   // 30 (minutes restantes une fois les heures retirées)

duree = duree.plusMinutes(30);   // Duration est immuable : il faut récupérer le résultat
Duration est immuable : plusXXX/minusXXX renvoient une nouvelle instance sans modifier l'objet d'origine. Il faut donc toujours réaffecter : duree = duree.plusMinutes(30);

Clonage d'objets

Cloner un objet signifie en créer une copie indépendante : modifier le clone ne doit pas affecter l'original, et inversement.

Rendre une classe clonable

  • Implémenter Cloneable — interface marqueur (aucune méthode déclarée).
  • Redéfinir clone(), héritée d'Object : protected à l'origine, à repasser en public.

protected Object Object.clone() throws CloneNotSupportedException

Lance CloneNotSupportedException (exception vérifiée) si la classe n'implémente pas Cloneable.

public class Boisson implements Cloneable {
    private String nom;

    @Override
    public Boisson clone() {
        try {
            return (Boisson) super.clone();
        } catch (CloneNotSupportedException e) {
            throw new AssertionError(e);
        }
    }
}
Comme la classe implémente Cloneable, CloneNotSupportedException ne peut jamais se produire en pratique : on l'attrape et on la convertit en AssertionError (non vérifiée) plutôt que de déclarer throws sur clone().

Copie superficielle (shallow copy)

super.clone() crée un nouvel objet et copie la valeur de chaque champ telle quelle. Suffisant pour un champ primitif ou un type immuable (String, LocalDate, Duration…). Pour un champ objet mutable ou une collection, seule la référence est copiée : original et clone pointent alors vers le même objet interne.

public class Commande implements Cloneable {
    private List<String> articles = new ArrayList<>();

    @Override
    public Commande clone() {
        try {
            return (Commande) super.clone();
            // ATTENTION : le clone et l'original partagent la même liste articles !
        } catch (CloneNotSupportedException e) {
            throw new AssertionError(e);
        }
    }
}
Avec cette copie superficielle, ajouter un article via le clone modifie aussi la liste de l'original — ce n'est pas le comportement attendu d'un clone.

Copie profonde (deep copy)

Après super.clone(), il faut remplacer chaque champ mutable par une copie indépendante :

@Override
public Commande clone() {
    try {
        Commande copie = (Commande) super.clone();
        copie.articles = new ArrayList<>(this.articles);   // nouvelle liste, mêmes éléments
        return copie;
    } catch (CloneNotSupportedException e) {
        throw new AssertionError(e);
    }
}

Si les éléments de la collection sont eux-mêmes mutables, il faut aussi les cloner un par un — copier la collection ne copie pas les objets qu'elle contient :

copie.articles = new ArrayList<>();
for (Article article : this.articles) {
    copie.articles.add(article.clone());
}
clone() doit dupliquer chaque champ mutable (listes, ensembles, objets internes mutables), pas se contenter du super.clone() par défaut.

Pièges fréquents

Problème Cause probable Correction
Doublons dans un HashSet equals/hashCode non redéfinis. Redéfinir les deux ensemble.
Élément « disparu » d'un TreeSet Comparaison qui renvoie 0 pour deux éléments différents. Rendre le Comparator/compareTo cohérent avec equals (départager sur tous les attributs).
UnsupportedOperationException sur un add Collection non modifiable (List.of, unmodifiableList…). Créer une copie modifiable : new ArrayList<>(collection).
NullPointerException sur un calcul Unboxing d'un wrapper null (ex. map.get sur clé absente). getOrDefault ou test de nullité avant l'unboxing.
ConcurrentModificationException Modification de la collection pendant un foreach. Utiliser un Iterator et sa méthode remove, ou removeIf.
Ordre inattendu à l'affichage HashSet/HashMap ne garantissent aucun ordre. Utiliser TreeSet/TreeMap (tri) ou LinkedHashSet/LinkedHashMap (ordre d'insertion).
Durée inchangée après plusMinutes/minusMinutes Duration est immuable : l'appel ne modifie pas l'objet, il en renvoie un nouveau. Réaffecter le résultat : duree = duree.plusMinutes(30);
Le clone modifie aussi l'original Copie superficielle (super.clone()) sur un champ mutable : la référence est partagée. Copie profonde : dupliquer chaque champ mutable après super.clone().
